4. 本项目的"签到脚本调度"场景小而稳,3.x 已足够;盲目追新只会自找苦吃

**升级时机: 等 4.0 GA 至少 6 个月、生态完成迁移后再评估**

### 1.2 Scheduler 类型选择 + 集成模式

**推荐架构: FastAPI lifespan + AsyncIOScheduler 同进程内运行**

```
┌────────────────────────────────────────────┐
│ FastAPI (uvicorn 单 worker)                │
│  ├─ 路由 /api/...                         │
│  ├─ SSE 路由 /api/logs/stream             │
│  └─ AsyncIOScheduler (同 event loop)      │
│      └─ 任务执行: 调用子进程跑签到脚本    │
└────────────────────────────────────────────┘
            ↓ jobstore
       SQLite(WAL 模式)
```

**三种方案对比:**

| 方案 | 优点 | 缺点 | 适用场景 |
|------|------|------|----------|
| **AsyncIOScheduler 同进程** ⭐ | 简单、共享 DB 连接、易访问 app 状态 | 重启 FastAPI 会重启调度器 | **本项目最佳** |
| BackgroundScheduler 同进程 | 任务跑在独立线程,不阻塞 API | 与 async 混用麻烦,需 `asyncio.run_coroutine_threadsafe` | 旧式同步代码 |
| 独立 worker 进程 | 解耦、可独立扩缩容 | 复杂度大幅上升,需 IPC/共享 DB,小项目过度设计 | 任务量极大或要高可用 |

**关键代码骨架:**

```python
# app/scheduler.py
from apscheduler.schedulers.asyncio import AsyncIOScheduler
from apscheduler.jobstores.sqlalchemy import SQLAlchemyJobStore
from apscheduler.executors.asyncio import AsyncIOExecutor
from apscheduler.executors.pool import ThreadPoolExecutor

def create_scheduler(db_url: str) -> AsyncIOScheduler:
    jobstores = {
        'default': SQLAlchemyJobStore(url=db_url)
    }
    executors = {
        # 异步任务用 AsyncIO 执行器
        'default': AsyncIOExecutor(),
        # 阻塞任务(子进程长时间等待)用线程池兜底
        'threadpool': ThreadPoolExecutor(max_workers=10),
    }
    job_defaults = {
        'coalesce': True,           # 错过多次合并为一次
        'max_instances': 1,         # 同一脚本同一时刻只跑一个实例 (核心!)
        'misfire_grace_time': 300,  # 错过 5 分钟内仍补跑
    }
    return AsyncIOScheduler(
        jobstores=jobstores,
        executors=executors,
        job_defaults=job_defaults,
        timezone='Asia/Shanghai',
    )

# app/main.py
from contextlib import asynccontextmanager
from fastapi import FastAPI, Request, Depends

@asynccontextmanager
async def lifespan(app: FastAPI):
    # startup
    scheduler = create_scheduler('sqlite:///./data/scheduler.db')
    scheduler.start()
    app.state.scheduler = scheduler
    yield
    # shutdown - wait=True 让正在跑的任务尽量收尾
    scheduler.shutdown(wait=True)

app = FastAPI(lifespan=lifespan)

# 路由中通过依赖注入访问
def get_scheduler(request: Request) -> AsyncIOScheduler:
    return request.app.state.scheduler

@app.post("/api/scripts/{script_id}/schedule")
async def schedule_script(
    script_id: int,
    cron: str,
    scheduler: AsyncIOScheduler = Depends(get_scheduler),
):
    scheduler.add_job(
        run_script_job,
        trigger='cron',
        args=[script_id],
        id=f"script-{script_id}",
        replace_existing=True,
        **parse_cron(cron),
    )
    return {"ok": True}
```

### 1.3 SQLAlchemyJobStore + SQLite 并发要点

**结论: 必须开启 WAL 模式 + 设置 `check_same_thread=False`**

**关键事项:**
1. **WAL 模式**(Write-Ahead Logging): 让读不阻塞写、写不阻塞读,大幅缓解 SQLite 单写者瓶颈
2. **check_same_thread=False**: SQLite 默认禁止跨线程,FastAPI/APScheduler 多线程访问时必须关闭此限制
3. **WAL 不能跨网络文件系统**(NFS),容器内挂载本地磁盘卷即可
4. **建议给 jobstore 单独一个 SQLite 文件**,与业务数据 DB 分离,避免互相阻塞

**配置示例:**

```python
from sqlalchemy import create_engine, event

engine = create_engine(
    "sqlite:///./data/scheduler.db",
    connect_args={"check_same_thread": False, "timeout": 30},
    pool_pre_ping=True,
)

@event.listens_for(engine, "connect")
def _enable_sqlite_wal(dbapi_conn, _):
    cur = dbapi_conn.cursor()
    cur.execute("PRAGMA journal_mode=WAL;")
    cur.execute("PRAGMA synchronous=NORMAL;")  # WAL 下推荐 NORMAL,牺牲少量持久性换性能
    cur.execute("PRAGMA busy_timeout=30000;")  # 写锁超时 30s
    cur.close()

jobstores = {'default': SQLAlchemyJobStore(engine=engine)}
```

参考: [SQLite WAL 文档](https://sqlite.org/wal.html)

### 1.4 在 cron job 里安全调用 async 函数

**AsyncIOScheduler 直接接受 async 函数,无需特殊处理:**

```python
@scheduler.scheduled_job('cron', hour='*/2', id='daily_signin')
async def my_async_signin_task():
    async with httpx.AsyncClient() as client:
        await client.post("...")
```

**反模式 ❌**: 在 BackgroundScheduler 里用 `asyncio.run(coro())` — 每次会创建新 event loop,跟 FastAPI 主 loop 隔离,数据库连接池等共享资源全废。

### 1.5 避免一个慢任务卡死调度器

**5 道防线:**

1. **`max_instances=1`**: 同一任务实例不重叠,跑得太慢就跳过下一次触发
2. **`coalesce=True`**: 错过的多次合并为一次,避免堆积
3. **`misfire_grace_time`**: 限定补跑窗口,过期就丢弃
4. **任务级超时**: 子进程运行用 `asyncio.wait_for(...)` 强制超时(详见第 2 章)
5. **executor 隔离**: CPU 密集任务用 `ProcessPoolExecutor`,长 I/O 用线程池,防止占满 async loop

```python
scheduler.add_job(
    long_running_task,
    'interval',
    minutes=10,
    id='heavy_job',
    executor='threadpool',  # 路由到线程池,不阻塞 async loop
    max_instances=1,
    coalesce=True,
    misfire_grace_time=120,
)
```

## 二、Python 子进程隔离 + 超时熔断

### 2.1 subprocess vs asyncio.create_subprocess_exec

**结论: 用 `asyncio.create_subprocess_exec`,理由如下:**

| 特性 | subprocess.Popen | asyncio.create_subprocess_exec |
|------|------------------|-------------------------------|
| 阻塞 event loop | ❌ 是 | ✅ 否 |
| 流式 stdout | 需要线程辅助 | 原生 `await proc.stdout.readline()` |
| 与 SSE 集成 | 麻烦 | 直接 yield |
| 超时 | `proc.wait(timeout=...)` | `asyncio.wait_for(...)` |

本项目要把脚本输出实时推到 SSE 端,**必须用 asyncio 版本**。

### 2.2 超时强杀 + 不留僵尸的标准写法(Linux)

**核心要点(Linux/Ubuntu 24):**
1. `start_new_session=True` 让子进程成为**新会话首进程**,自动建立独立进程组(PGID = PID)
2. 超时后 `os.killpg(pgid, signal.SIGTERM)` 给整个进程组发信号(子进程的子进程也一并清理)
3. 给一个 grace period(如 5s),仍存活则升级为 `SIGKILL`
4. 始终 `await proc.wait()` 回收僵尸

```python
import asyncio
import os
import signal
from typing import AsyncIterator

async def run_script_streaming(
    cmd: list[str],
    cwd: str,
    timeout: float = 600,
    env: dict | None = None,
) -> AsyncIterator[tuple[str, str]]:
    """
    跑子进程 + 流式 yield (stream_name, line),
    支持超时强杀(进程组),不留僵尸。
    yield ('stdout', line) / ('stderr', line) / ('exit', returncode)
    """
    proc = await asyncio.create_subprocess_exec(
        *cmd,
        cwd=cwd,
        env=env,
        stdout=asyncio.subprocess.PIPE,
        stderr=asyncio.subprocess.PIPE,
        # 关键: Linux 上让子进程独立成会话/进程组首
        start_new_session=True,
    )

    async def read_stream(stream, name):
        while True:
            line = await stream.readline()
            if not line:
                return
            yield (name, line.decode('utf-8', errors='replace').rstrip())

    async def merge():
        # 简化版: 用 asyncio.Queue 合并两路输出
        q: asyncio.Queue = asyncio.Queue()

        async def pump(reader, name):
            async for item in read_stream(reader, name):
                await q.put(item)
            await q.put(None)  # EOF 标记

        t1 = asyncio.create_task(pump(proc.stdout, 'stdout'))
        t2 = asyncio.create_task(pump(proc.stderr, 'stderr'))

        eofs = 0
        while eofs < 2:
            item = await q.get()
            if item is None:
                eofs += 1
                continue
            yield item
        await t1; await t2

    try:
        async for item in merge_with_timeout(merge(), proc, timeout):
            yield item
    finally:
        # 兜底回收
        if proc.returncode is None:
            await _kill_process_group(proc)
        await proc.wait()
        yield ('exit', str(proc.returncode))


async def merge_with_timeout(gen, proc, timeout):
    """对生成器整体设置超时,触发后强杀进程组"""
    try:
        async with asyncio.timeout(timeout):  # Python 3.11+
            async for item in gen:
                yield item
    except TimeoutError:
        yield ('stderr', f'[TIMEOUT after {timeout}s, killing process group]')
        await _kill_process_group(proc)


async def _kill_process_group(proc: asyncio.subprocess.Process):
    """先 SIGTERM 给进程组,5s 后 SIGKILL"""
    if proc.returncode is not None:
        return
    try:
        pgid = os.getpgid(proc.pid)
        os.killpg(pgid, signal.SIGTERM)
        try:
            await asyncio.wait_for(proc.wait(), timeout=5)
            return
        except asyncio.TimeoutError:
            os.killpg(pgid, signal.SIGKILL)
            await proc.wait()
    except (ProcessLookupError, PermissionError):
        pass
```

### 2.3 Docker 容器内是否还需要 ulimit?

**简答: 一般不需要,但建议给 compose 加几条防御性配置:**

```yaml
services:
  backend:
    # 防 fork bomb / 内存炸弹
    ulimits:
      nproc: 4096
      nofile: 65536
    mem_limit: 1g           # 整容器内存上限 (Compose v3+ 用 deploy.resources)
    pids_limit: 1024        # 防止疯狂创建子进程
```

子进程是 Python 脚本,失控概率小,**最重要的是任务级超时(2.2)+ 容器级 mem_limit / pids_limit**。

### 2.4 环境隔离

签到脚本可能依赖各种 Python 包,**强烈建议每个脚本一个独立 venv**(或 uv-managed env),不要往主进程 venv 里堆三方依赖。

```
scripts/
  ├─ bilibili-daily/
  │   ├─ main.py
  │   ├─ requirements.txt
  │   └─ .venv/            ← 独立 venv
  └─ v2ex-daily/
      └─ ...
```

执行命令: `["./scripts/bilibili-daily/.venv/bin/python", "main.py"]`

### 3.2 异步使用(`async_notify`)

```python
import asyncio
from apprise import Apprise, AppriseAsset, NotifyType

asset = AppriseAsset(app_id="签到面板", app_desc="签到脚本聚合管理")
ap = Apprise(asset=asset)

# 添加多个渠道,失败一个不影响其他
ap.add('tgram://BOT_TOKEN/CHAT_ID/?format=markdown')
ap.add('wecombot://BOT_KEY')

async def notify(title: str, body: str, success: bool):
    notify_type = NotifyType.SUCCESS if success else NotifyType.FAILURE
    # async_notify 是协程,返回 bool (全部成功才 True)
    return await ap.async_notify(
        title=title,
        body=body,
        notify_type=notify_type,
        body_format='markdown',
    )

# 在 FastAPI / APScheduler 里直接 await
asyncio.run(notify("签到完成", "成功 5 个 / 失败 1 个", True))
```

**注意点:**
- `async_notify()` 内部会并发推送给所有添加的 endpoint
- 单个渠道失败不会抛异常,只反映在返回值;需要逐个状态可用 `notify(... return_status_objects=True)`
- 推送渠道 URL 中的密钥/token **必须**走配置加密(详见第 7 章)

## 五、SSE 实时日志推送(FastAPI 端)

### 5.1 库选择: sse-starlette 仍是首选

**当前最新: sse-starlette 3.4.4(2026-05-12 发布)**

虽然 FastAPI 0.135.0+ 内置了 `fastapi.sse.EventSourceResponse`(本质就是 sse-starlette 的封装),**但 sse-starlette 独立版本仍然更灵活**(自定义 ping、shutdown grace period、事件 ID 控制),推荐直接用 sse-starlette。

```bash
pip install sse-starlette
```

### 5.2 子进程 stdout → SSE 桥接(完整骨架)

```python
# app/api/logs.py
from fastapi import APIRouter, Request, Depends
from sse_starlette import EventSourceResponse, ServerSentEvent
import asyncio
import json
import time

router = APIRouter()

@router.get("/api/v1/runs/{run_id}/logs/stream")
async def stream_run_logs(run_id: int, request: Request):
    async def event_generator():
        try:
            async for stream_name, line in run_log_broker.subscribe(run_id):
                # 客户端断开就停
                if await request.is_disconnected():
                    break

                yield ServerSentEvent(
                    data=json.dumps({
                        "stream": stream_name,
                        "line": line,
                        "ts": time.time(),
                    }),
                    event="log",
                )
        except Exception as e:
            yield ServerSentEvent(
                data=json.dumps({"error": str(e)}),
                event="error",
            )
        finally:
            yield ServerSentEvent(data="done", event="end")

    return EventSourceResponse(
        event_generator(),
        ping=15,                       # 15s 心跳保活
        send_timeout=5,                # 单次 send 超时
    )
```

### 5.3 历史日志回放 + 实时跟随的双场景

签到面板典型需求:
- 任务跑完后用户进来要看**历史完整日志** (REST,不用 SSE)
- 任务正在跑的时候进来要**实时跟随**(SSE)

推荐做法:
1. 子进程 stdout 进入一个内存 `asyncio.Queue` + 同步落盘到 `runs.stdout` 累积 buffer
2. SSE 端口先把已有日志一次性 yield(从 DB 读),再切到 Queue 实时跟随
3. 任务结束后 SSE 端口 yield 完缓冲就关流

```python
async def event_generator():
    # 阶段 1: 回放已有日志
    run = await get_run(run_id)
    if run.stdout:
        for line in run.stdout.splitlines():
            yield ServerSentEvent(data=line, event="log")

    # 阶段 2: 跟随实时输出
    queue = log_broker.get_queue(run_id)
    if queue is None:
        # 任务已经跑完
        yield ServerSentEvent(
            data=json.dumps({"status": run.status, "exit_code": run.exit_code}),
            event="status",
        )
        yield ServerSentEvent(data="finished", event="end")
        return

    while True:
        if await request.is_disconnected():
            break
        try:
            item = await asyncio.wait_for(queue.get(), timeout=30)
        except asyncio.TimeoutError:
            continue  # 让 ping 心跳保活
        if item is None:  # 终止信号
            break
        yield ServerSentEvent(data=item, event="log")
```

### 5.4 客户端断开检测 & 心跳保活

- **断开检测**: 必须在循环里 `if await request.is_disconnected(): break`,否则子进程一直在跑、Queue 一直在塞、内存爆炸
- **心跳保活**: sse-starlette 默认每 15s 发一条 `: ping` 注释行,大多数场景够用;Nginx/Caddy 反代下重要(防止代理超时关连接)

### 5.5 Nginx/Caddy 反代 SSE 的必备配置

**Nginx**(如果用的话):
```nginx

    proxy_pass http://backend;
    proxy_http_version 1.1;
    proxy_set_header Connection "";
    proxy_buffering off;            # 关键!关闭缓冲
    proxy_cache off;
    proxy_read_timeout 24h;          # 长连接
    chunked_transfer_encoding off;
}
```

**Caddy**(推荐,自动正确):

```
    }

```

## 六、Ubuntu 24 + Docker Compose 部署

### 6.1 镜像结构: 后端 + 前端 caddy 分开

**推荐架构(两个服务):**

```
┌─────────────┐    ┌─────────────┐
│   caddy     │←──→│  backend    │
│  (反代+TLS) │    │  (FastAPI)  │
│  + serve    │    │             │
│  前端 dist  │    │             │
└─────────────┘    └─────────────┘
       ↑                   ↑
    用户 HTTPS         scripts/, data/
                       (host 挂载)
```

为啥这样?
- 前端是纯静态,可以让 caddy 直接 serve `dist/`,**不需要单独 nginx 镜像**
- backend 单独跑 FastAPI/uvicorn,镜像里有 Python 环境
- caddy 处理反代 + 自动 HTTPS,无 cron / certbot 之痛

### 6.2 反代选型: **Caddy 强烈推荐(2026)**

| 特性 | Caddy 2.x | Nginx + Certbot |
|------|-----------|-----------------|
| 自动 HTTPS | ✅ 内置 ACME,零配置 | ❌ 需 certbot/acme.sh + cron |
| 配置文件 | Caddyfile,人性化 | nginx.conf,语法繁 |
| 反代 SSE 默认行为 | ✅ 默认正确(可加 flush_interval -1 强化) | ❌ 默认开缓冲,需手动关 |
| HTTP/2、HTTP/3 | ✅ 默认 | 需配置 |
| 内存占用 | 25-35MB(固定) | 5-10MB(更小) |
| 性能 | 略低于 Nginx | 顶尖 |
| 适合本项目 | ⭐⭐⭐⭐⭐ | ⭐⭐⭐ |

**结论**: 小团队、单服务器、签到面板这种规模,Caddy 完胜。Nginx 适合大流量、有专职运维的项目。

### 6.3 卷挂载权限 + UID/GID 映射

## 七、配置加密

### 7.1 cryptography.fernet 仍是首选

**结论: Fernet 对小型项目仍是 2026 年的合理选择**

**Fernet 算法细节:**
- AES-128-CBC 加密
- HMAC-SHA256 认证
- 内部生成 IV(`os.urandom(16)`)
- 输出是 URL-safe base64 编码的"token"

**不适合**: 超大文件(整个明文需要在内存里)、需要跨语言互操作的场景。
**适合**: 配置项加密、token 加密、数据库字段加密。

**当前推荐版本: cryptography 44.0.0+(2026)**

```bash
pip install cryptography
```

### 7.2 主密钥管理(实用建议)

**小团队的现实方案(按推荐度排序):**

1. **本地 keyfile + 严格权限**(本项目推荐)
   ```bash
   # 生成
   python -c "from cryptography.fernet import Fernet; print(Fernet.generate_key().decode())" > data/encryption.key
   chmod 600 data/encryption.key
   chown 1000:1000 data/encryption.key
   ```
   - 优点: 简单,不依赖外部服务
   - **关键**: encryption.key 必须**离线异地备份**(密码管理器、打印纸保险柜)

2. **环境变量**(开发/CI 友好)
   ```yaml
   environment:
     - ENCRYPTION_KEY=${ENCRYPTION_KEY}
   ```
   - 适合 CI 注入,不写入镜像
   - 仍需要外部 secret store(GitHub Actions Secrets / 1Password CLI)

3. **HashiCorp Vault / Bitwarden Secrets Manager**(团队规模大时)
   - 过度设计;签到面板大概率不需要

**密钥轮换(MultiFernet):**

```python
from cryptography.fernet import Fernet, MultiFernet

# 新密钥放前面,旧密钥保留兼容历史 token
new_key = Fernet(load_key('encryption.key.new'))
old_key = Fernet(load_key('encryption.key'))
mf = MultiFernet([new_key, old_key])

# 加密用 new_key
token = mf.encrypt(b"secret")

# 解密自动尝试列表中所有密钥
plaintext = mf.decrypt(token)

# 重新加密所有历史 token (rotate)
new_token = mf.rotate(old_token)
```

**轮换流程**:
1. 生成新密钥 → 添加到 MultiFernet 列表前部
2. 后台任务遍历所有加密字段,调用 `mf.rotate(old_token)` 重新加密
3. 全部迁移完成后,从列表中移除旧密钥

### 7.3 字段级 vs 整行加密

**结论: 字段级加密**(只加密敏感字段,如密码、token、cookie 等)

| 维度 | 字段级 | 整行加密 |
|------|--------|---------|
| 查询能力 | 非加密字段可索引、JOIN | 完全失去查询能力 |
| 性能 | 仅敏感列开销 | 每次读写都要解密 |
| 实现复杂度 | 用 SQLAlchemy 自定义 TypeDecorator | 简单 |
| 安全性 | 高(字段级,关键即可) | 高 |

**SQLAlchemy 字段级加密示例:**

```python
# app/db/types.py
from sqlalchemy.types import TypeDecorator, Text
from cryptography.fernet import Fernet, MultiFernet

class EncryptedText(TypeDecorator):
    impl = Text
    cache_ok = True

    def __init__(self, fernet: MultiFernet, **kw):
        super().__init__(**kw)
        self._fernet = fernet

    def process_bind_param(self, value, dialect):
        if value is None:
            return None
        return self._fernet.encrypt(value.encode('utf-8')).decode('ascii')

    def process_result_value(self, value, dialect):
        if value is None:
            return None
        return self._fernet.decrypt(value.encode('ascii')).decode('utf-8')
```

### 7.4 加密最佳实践速查

| 实践 | 说明 |
|------|------|
| ✅ 加密配置中的 cookie/token/password | 这些是签到脚本的核心敏感数据 |
| ✅ encryption.key 文件权限 600 | 防止其他用户读取 |
| ✅ encryption.key 异地离线备份 | 主密钥丢 = 所有加密数据废 |
| ✅ 用 MultiFernet 准备好轮换能力 | 即使现在不轮换,代码结构也要支持 |
| ❌ 不要把 encryption.key 写进镜像 | 必须运行时挂载 |
| ❌ 不要把 encryption.key 入 git | .gitignore 必须包含 |
| ❌ 不要在日志里打印密钥/明文 token | 日志脱敏 |

- [cryptography Fernet 官方文档](https://cryptography.io/en/latest/fernet/)

## 总结决策表

| 决策点 | 选择 | 关键理由 |
|--------|------|---------|
| 调度器版本 | APScheduler **3.11.2** | 4.0 仍 alpha,不能用 |
| 调度器类型 | **AsyncIOScheduler** + FastAPI lifespan | 与 async 生态一致,共享 event loop |
| 调度器 jobstore | **MemoryJobStore**(事实源是 instances 表) | 简单可靠,不与业务库竞争锁 |
| 子进程 | `asyncio.create_subprocess_exec` + `start_new_session=True` + `os.killpg` | async 友好,流式 stdout,超时强杀进程组 |
| 通知库 | **apprise 1.10+** + 自己加 jinja2 模板层 | 80+ 渠道,中文渠道全支持 |
| 前端脚手架 | **create-vue** + Vite + TypeScript | 官方推荐 |
| UI 库 | **Element Plus 2.14+** + 按需引入 + dark mode CSS vars | 中文生态最佳 |
| API 类型生成 | **openapi-typescript + openapi-fetch** | 轻量、零运行时、官方维护 |
| 状态管理 | Pinia + pinia-plugin-persistedstate | Vue 3 标配 |
| SSE 实现 | **sse-starlette 3.4+** | 最成熟、最灵活 |
| 反向代理 | **Caddy 2** | 自动 HTTPS,反代 SSE 默认正确 |
| 容器基础镜像 | python:3.12-slim 多阶段构建 | distroless 调试苦,slim 平衡好 |
| 容器用户 | 固定 UID/GID=1000 + Dockerfile USER + tini | 解决卷挂载权限 |
| 备份 | restic + cron + 远端仓库 | 加密、增量、去重 |
| 配置加密 | **cryptography.Fernet + MultiFernet** + 字段级 | 简单、足够安全、可轮换 |

## 给编码 agent 的实施建议

1. **第一步**: 用 create-vue 起前端项目,用 uv 起后端项目,先把"hello world"打通(Caddy 反代 + FastAPI + Vue 三件套都跑起来)
2. **第二步**: 把 SQLAlchemy 模型 + Fernet 字段加密层做好,建立 SQLite WAL 配置
3. **第三步**: 实现子进程 runner(2.2 章节代码),先单元测试超时强杀
4. **第四步**: 接入 APScheduler,实现"添加任务/触发任务/查看任务列表"
5. **第五步**: SSE 日志实时推送,前端 EventSource 跟随
6. **第六步**: apprise 通知,模板化封装
7. **第七步**: Docker Compose 生产部署 + restic 备份脚本

每一步都先跑通最小可用版本,再加细节。**忌讳一次性写大量代码**。
