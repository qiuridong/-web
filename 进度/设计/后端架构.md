# 签到脚本聚合管理面板 — 后端详细架构设计文档

> 版本:v1.0
> 适用范围:个人/小团队自用,Ubuntu 24 + Docker Compose 单机部署
> 目标读者:后端编码 agent、前端联调 agent、运维 agent
> 设计原则:**轻量、可读、可演进**,SQLite 单文件可备份,插件零侵入,默认安全

---

## 0. 文档约定

- 字段命名一律 `snake_case`;HTTP 路径一律 `kebab-case`;模型类名 `PascalCase`
- 所有时间字段统一存 **UTC**(`DateTime(timezone=True)`),前端按用户时区呈现
- 所有金额/计数类字段统一 `Integer` 或 `BigInteger`,布尔类用真正的 `Boolean`
- "加密 blob"指经 Fernet 加密后的 base64 字符串(URL-safe),数据库存 `Text`
- API 文档中标注 `🔒` 表示需要鉴权,`🔓` 表示匿名可访问

## 1. 数据模型

### 1.1 总览(ER 关系)

```
users (1) ── 仅做登录认证,与业务表无外键

scripts (1) ──< (N) instances (1) ──< (N) runs
                       │
                       └──< (N) notification_rules >── (N) notification_channels

notification_rules ── 可绑定 global / script / instance 三种 scope
settings ── 全局 KV,无外键
```

### 1.2 表 `scripts`(脚本插件元信息)

由"扫描 `scripts/` 目录"动作写入/更新,**不存储脚本源码本身**,只存可被前端展示和调度引用的元数据。

| 字段 | 类型 | 约束 | 说明 |
|---|---|---|---|
| `id` | `Integer` | PK, autoincrement | 内部主键 |
| `slug` | `String(64)` | unique, not null, indexed | 脚本唯一标识,等同目录名,如 `bilibili-daily` |
| `name` | `String(128)` | not null | 展示名,如"B站每日签到" |
| `description` | `Text` | nullable | 功能简介 |
| `version` | `String(32)` | not null, default `'0.0.0'` | 脚本版本,支持版本变更告警 |
| `author` | `String(64)` | nullable | 作者 |
| `homepage` | `String(256)` | nullable | 主页链接 |
| `default_cron` | `String(64)` | nullable | 默认调度表达式,如 `0 8 * * *` |
| `default_timeout_sec` | `Integer` | not null, default `300` | 默认超时秒数 |
| `fields_schema_json` | `JSON`(SQLite 用 `Text` + JSON1) | not null, default `'[]'` | 字段定义数组,见 §3.1 |
| `requires_secret` | `Boolean` | not null, default `false` | 是否包含 secret 字段(冗余,便于列表筛选) |
| `enabled` | `Boolean` | not null, default `true` | 全局禁用后所有实例都不会被调度 |
| `manifest_path` | `String(512)` | not null | manifest.yaml 绝对路径,用于热重载 |
| `manifest_hash` | `String(64)` | not null | manifest 内容 SHA256,用于检测变更 |
| `last_scanned_at` | `DateTime(tz)` | not null | 最近一次扫描入库时间 |
| `created_at` | `DateTime(tz)` | not null, default `now()` | |
| `updated_at` | `DateTime(tz)` | not null, onupdate `now()` | |

**索引**
- `ix_scripts_slug`(unique)
- `ix_scripts_enabled`

**关系**
- `instances`: `relationship("Instance", back_populates="script", cascade="all, delete-orphan")`

**说明**
- 删除 `scripts` 行 = 从 DB 移除该插件登记,**不删磁盘文件**;下次扫描会重新发现
- `fields_schema_json` 是从 manifest.yaml `fields:` 段解析得到的快照,实例创建时按此 schema 校验

### 1.3 表 `instances`(脚本实例)

一个脚本可创建多个实例(典型场景:同一脚本管多个账号)。这是调度的基本单位。

| 字段 | 类型 | 约束 | 说明 |
|---|---|---|---|
| `id` | `Integer` | PK, autoincrement | |
| `script_id` | `Integer` | FK → `scripts.id` ON DELETE CASCADE, not null, indexed | |
| `name` | `String(128)` | not null | 用户起的别名,如"B站-主号" |
| `description` | `String(256)` | nullable | 备注 |
| `cron_expr` | `String(64)` | nullable | 留空则继承 `scripts.default_cron` |
| `timeout_sec` | `Integer` | nullable | 留空则继承 `scripts.default_timeout_sec` |
| `enabled` | `Boolean` | not null, default `true` | 控制是否参与调度 |
| `paused_until` | `DateTime(tz)` | nullable | 临时暂停到某时刻,过后自动恢复 |
| `config_blob` | `Text` | not null, default `''` | Fernet 加密后的 JSON,见 §5.2 |
| `config_version` | `Integer` | not null, default `1` | 加密 schema 版本号,便于将来 key rotation |
| `max_retries` | `Integer` | not null, default `0` | 失败自动重试次数 |
| `retry_interval_sec` | `Integer` | not null, default `60` | 首次重试等待秒数,后续按指数退避 |
| `last_run_id` | `Integer` | FK → `runs.id` ON DELETE SET NULL, nullable | 冗余指向最近一次执行,便于列表免 join |
| `last_run_status` | `String(16)` | nullable | `success` / `failure` / `error` / `timeout` 冗余,用于列表筛选 |
| `last_run_at` | `DateTime(tz)` | nullable | 冗余,排序常用 |
| `next_run_at` | `DateTime(tz)` | nullable | 由调度器写入,UI 展示"下次执行" |
| `total_runs` | `Integer` | not null, default `0` | 累计执行次数,统计用 |
| `total_successes` | `Integer` | not null, default `0` | 累计成功次数 |
| `created_at` | `DateTime(tz)` | not null, default `now()` | |
| `updated_at` | `DateTime(tz)` | not null, onupdate `now()` | |

**索引**
- `ix_instances_script_id`
- `ix_instances_enabled`
- `ix_instances_next_run_at`(便于"即将执行"看板)
- 复合:`ix_instances_script_enabled (script_id, enabled)`

**关系**
- `script`: `relationship("Script", back_populates="instances")`
- `runs`: `relationship("Run", back_populates="instance", cascade="all, delete-orphan", order_by="Run.started_at.desc()")`

**说明**
- `config_blob` 解密后是 dict,key 与 `scripts.fields_schema_json` 中字段定义对应
- `last_run_*` 三个冗余字段在每次 run 结束后由调度器同步,避免列表页 N+1 join
- 删除 instance 时级联删除其所有 runs

### 1.4 表 `runs`(执行历史)

**最大、写入最频繁的表**,索引设计要慎重。

| 字段 | 类型 | 约束 | 说明 |
|---|---|---|---|
| `id` | `Integer` | PK, autoincrement | |
| `instance_id` | `Integer` | FK → `instances.id` ON DELETE CASCADE, not null, indexed | |
| `script_slug` | `String(64)` | not null, indexed | 冗余:即便 instance 删了便于按脚本统计;也用于不 join 的快速筛选 |
| `trigger_type` | `String(16)` | not null | `manual` / `scheduled` / `retry` / `api` |
| `trigger_user_id` | `Integer` | FK → `users.id` ON DELETE SET NULL, nullable | manual 触发时记录是谁点的 |
| `parent_run_id` | `Integer` | FK → `runs.id`, nullable | retry 时指向上一次失败的 run |
| `status` | `String(16)` | not null, indexed | `pending` / `running` / `success` / `failure` / `error` / `timeout` / `cancelled` |
| `exit_code` | `Integer` | nullable | 子进程退出码,timeout 时为 `-15` 等 |
| `result_message` | `String(512)` | nullable | 脚本返回的 RunResult.message,失败原因摘要 |
| `result_data_json` | `Text` | nullable | RunResult.data 的 JSON,通常很短 |
| `stdout` | `Text` | nullable | 完整 stdout,大日志走压缩(见说明) |
| `stderr` | `Text` | nullable | 完整 stderr |
| `stdout_truncated` | `Boolean` | not null, default `false` | 超出 `max_log_bytes` 时截断,标记便于前端提示 |
| `stderr_truncated` | `Boolean` | not null, default `false` | |
| `started_at` | `DateTime(tz)` | not null, indexed | 开始时间 |
| `finished_at` | `DateTime(tz)` | nullable | 结束时间;`pending`/`running` 时为空 |
| `duration_ms` | `Integer` | nullable | 冗余,= `(finished_at - started_at).total_seconds() * 1000`,排序/统计快 |
| `host` | `String(64)` | nullable | 执行机标识,默认取 `socket.gethostname()`,为将来分布式预留 |
| `created_at` | `DateTime(tz)` | not null, default `now()` | |

**索引策略(关键)**

执行历史的高频查询模式:
1. 某 instance 的最近 N 条 → 复合索引 `(instance_id, started_at DESC)`
2. 全局最近 N 条 → 单列索引 `started_at DESC`
3. 失败筛选 → `(status, started_at DESC)`
4. 按脚本聚合统计 → `(script_slug, started_at DESC)`
5. 清理旧记录 → `started_at` 已有索引可复用

具体声明:
- `ix_runs_instance_started`(`instance_id`, `started_at` DESC)
- `ix_runs_started_at`(`started_at` DESC)
- `ix_runs_status_started`(`status`, `started_at` DESC)
- `ix_runs_script_started`(`script_slug`, `started_at` DESC)

**说明**
- `stdout`/`stderr` 各设上限,默认 `max_log_bytes = 256 KiB`,超出尾部截断,前端提示"日志过大,已截断,请查看 sse 实时日志"
- `result_data_json` 不应放大量数据,脚本若需输出大数据请走 stdout
- `pending` 状态设计原因:从"调度器创建 run 行"到"子进程真正起来"之间存在毫秒级窗口,此期间状态为 pending;子进程启动后改为 running

### 1.6 表 `notification_rules`(通知规则)

| 字段 | 类型 | 约束 | 说明 |
|---|---|---|---|
| `id` | `Integer` | PK, autoincrement | |
| `name` | `String(64)` | not null | 规则别名,便于管理 |
| `scope` | `String(16)` | not null | `global` / `script` / `instance` |
| `script_id` | `Integer` | FK → `scripts.id` ON DELETE CASCADE, nullable, indexed | scope=script/instance 时必填 script_id |
| `instance_id` | `Integer` | FK → `instances.id` ON DELETE CASCADE, nullable, indexed | scope=instance 时必填 |
| `event` | `String(16)` | not null | `success` / `failure` / `error` / `timeout` / `any` |
| `channel_id` | `Integer` | FK → `notification_channels.id` ON DELETE CASCADE, not null, indexed | |
| `template` | `Text` | nullable | 自定义模板,留空走默认模板 |
| `min_interval_sec` | `Integer` | not null, default `0` | 同规则两次通知最小间隔,防刷屏(0 = 不限) |
| `last_fired_at` | `DateTime(tz)` | nullable | 用于 `min_interval_sec` 节流判断 |
| `enabled` | `Boolean` | not null, default `true` | |
| `created_at` | `DateTime(tz)` | not null, default `now()` | |
| `updated_at` | `DateTime(tz)` | not null, onupdate `now()` | |

**约束**
- 应用层校验:`scope=script` 时 `script_id` 必填、`instance_id` 必空;`scope=instance` 时两者都必填且 `instance.script_id == script_id`;`scope=global` 时两者都必空
- 复合索引 `ix_rules_scope_lookup (scope, script_id, instance_id, event, enabled)`,匹配时一次扫描

**关系**
- `channel`: `relationship("NotificationChannel", back_populates="rules")`
- `script`: `relationship("Script")`(单向)
- `instance`: `relationship("Instance")`(单向)

**匹配优先级**(同一事件可能命中多条规则)
- 默认所有命中且 enabled 的规则都触发各自渠道(用户语义:多渠道并发通知)
- 若多条规则命中**同一渠道**,取 scope 更具体的一条(instance > script > global)以避免重复推送

## 2. REST API 路由清单

- 路由前缀:`/api/v1`
- 默认响应格式:`application/json; charset=utf-8`
- 鉴权方式:HttpOnly Cookie(详见 §5.3),所有 🔒 端点要求有效 session
- 错误响应格式见 §8.2
- 分页参数统一:`page`(从 1 起,默认 1)、`page_size`(默认 20,最大 200)
- 列表响应统一:`{ "items": [...], "total": N, "page": p, "page_size": ps }`

### 2.1 鉴权 `/api/v1/auth`

| Method | Path | 鉴权 | 用途 |
|---|---|---|---|
| GET | `/auth/setup-status` | 🔓 | 返回 `{ "needs_setup": bool }`,前端用以决定首屏跳"设置密码" |
| POST | `/auth/setup` | 🔓 | 首次创建管理员;**仅当用户表为空时可调用**;请求 `{ username, password, display_name? }`;成功后自动登录 |
| POST | `/auth/login` | 🔓 | `{ username, password }`;成功 → 设置 session cookie,返回 `{ user }` |
| POST | `/auth/logout` | 🔒 | 销毁 session;204 |
| GET | `/auth/me` | 🔒 | 返回当前用户 `{ id, username, display_name, is_admin, last_login_at }` |
| POST | `/auth/change-password` | 🔒 | `{ old_password, new_password }`;成功后强制重新登录 |

### 2.2 脚本 `/api/v1/scripts`

| Method | Path | 鉴权 | 用途 |
|---|---|---|---|
| GET | `/scripts` | 🔒 | 列表,可选 `enabled`、`q`(模糊匹配 name/slug)筛选;响应字段含 `id, slug, name, description, version, default_cron, enabled, instance_count, last_scanned_at` |
| GET | `/scripts/{slug}` | 🔒 | 详情,含 `fields_schema`(已解析数组)、`requirements_present`(bool)、`readme_md`(若有)、`icon_url`(若有) |

| POST | `/scripts/{slug}/enable` | 🔒 | 全局启用 |
| POST | `/scripts/{slug}/disable` | 🔒 | 全局禁用,会暂停其所有实例的调度但不删 |
| DELETE | `/scripts/{slug}` | 🔒 | 从 DB 移除登记(不删磁盘),会级联删除其所有 instance 与 run;请求需带 `?confirm=true` |

### 2.3 实例 `/api/v1/instances`

| Method | Path | 鉴权 | 用途 |
|---|---|---|---|
| GET | `/instances` | 🔒 | 列表,可选 `script_slug`、`enabled`、`status`(取自 `last_run_status`)筛选;字段含 `id, name, script: {slug,name}, cron_expr, enabled, last_run_status, last_run_at, next_run_at, total_runs, total_successes` |
| POST | `/instances` | 🔒 | 创建;请求 `{ script_slug, name, description?, cron_expr?, timeout_sec?, max_retries?, retry_interval_sec?, config: {...} }`;config 按 fields_schema 严格校验,secret 字段服务端加密落库 |
| GET | `/instances/{id}` | 🔒 | 详情;**secret 字段返回 `null`**,但 `_secret_set: { field_name: bool }` 指示是否已配置过 |
| PATCH | `/instances/{id}` | 🔒 | 部分更新;只传需要改的字段;config 中**未提交的 secret 字段保持原值**(用于"不改密码只改其他") |

| POST | `/instances/{id}/enable` | 🔒 | 启用并(重新)注册到调度器 |
| POST | `/instances/{id}/disable` | 🔒 | 禁用并从调度器摘除 |

| POST | `/instances/{id}/resume` | 🔒 | 立即清除 `paused_until` |
| POST | `/instances/{id}/run` | 🔒 | 立即触发一次;返回 `{ run_id }`;若达并发上限则排队 |
| POST | `/instances/{id}/test` | 🔒 | 与 run 类似,但不写 runs 表、不发通知,纯试运行;响应包含完整 stdout/stderr/result;**不可与 run 同时运行同一 instance** |

### 2.4 执行历史 `/api/v1/runs`

| Method | Path | 鉴权 | 用途 |
|---|---|---|---|
| GET | `/runs` | 🔒 | 列表;可选 `instance_id`、`script_slug`、`status`、`trigger_type`、`started_after`、`started_before`;按 `started_at DESC` 排序;响应不含 stdout/stderr |
| GET | `/runs/{id}` | 🔒 | 详情;含完整 stdout/stderr/result,标识截断 |
| GET | `/runs/{id}/logs/stream` | 🔒 | **SSE 实时日志**,详见 §2.4.1 |
| POST | `/runs/{id}/cancel` | 🔒 | 取消正在运行的 run(向子进程发 SIGTERM,超时 5 秒后 SIGKILL);仅 `pending`/`running` 可取消 |
| DELETE | `/runs/cleanup` | 🔒 | 手动触发清理;`{ before: ISO8601 }` 或 `{ keep_days: N }`;返回 `{ deleted: N }` |

#### 2.4.1 SSE 端点详细设计 `GET /runs/{id}/logs/stream`

- 响应头 `Content-Type: text/event-stream`,`Cache-Control: no-cache`,`X-Accel-Buffering: no`(防 nginx 缓冲)
- 事件流:
  - `event: stdout` `data: <一行文本,UTF-8>`
  - `event: stderr` `data: <一行文本>`
  - `event: status` `data: {"status":"running"|"success"|"failure"|...,"exit_code":n,"duration_ms":m}`
  - `event: ping` `data: <unix_ts>` 每 15 秒一次,保活
  - 终止:发送一条 `event: end`,然后服务端关闭连接
- 实现要点:
  - 子进程 stdout/stderr 通过内存 broker(`asyncio.Queue` per run)广播给所有订阅者
  - 历史输出:连接时先回放已落库的 stdout/stderr 切片,再开始转发实时流
  - 若连接时 run 已结束,立即发送一次 status + end,然后断开
  - 单 run 订阅者数硬上限 10,超出返回 429

### 3.3 main.py 契约

每个脚本必须暴露 **顶层函数 `run`**,签名固定:

```text
def run(config: dict, context: RunContext) -> RunResult
```

#### 3.3.1 输入

- `config: dict` — 已解密、已按 fields_schema 解析的实例配置(secret 字段是明文)
- `context: RunContext` — 由主程序注入的运行时对象,字段如下:

| 字段 | 类型 | 说明 |
|---|---|---|
| `run_id` | int | 当前 run 的 DB 主键,可记入日志便于追踪 |
| `instance_id` | int | |
| `instance_name` | str | |
| `script_slug` | str | |
| `script_dir` | str | 脚本所在目录绝对路径,可用于读取脚本附属资源(如静态文件) |
| `data_dir` | str | 脚本专属数据目录(`data/scripts/<slug>/<instance_id>/`),由主程序创建,脚本可读写,**唯一允许写的位置** |
| `timeout_sec` | int | 本次执行的超时时间(脚本可用于 HTTP 请求超时设置) |
| `trigger_type` | str | `manual` / `scheduled` / `retry` / `api` |
| `attempt` | int | 第几次尝试,首次 = 1,retry 时 ≥ 2 |
| `logger` | Logger | 已配置的 logger,写入即出现在 stdout(详见 §8.4) |
| `notify(title, body, level='info')` | callable | 脚本内可主动推一条通知(走全局 channels,与执行结果通知独立) |

#### 3.3.2 输出 `RunResult`

是一个轻量 dataclass / Pydantic 模型,字段:

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| `success` | bool | 是 | true = success,false = failure |
| `message` | str | 否 | 简短摘要,会写入 `runs.result_message`,长度 ≤ 512 |
| `data` | dict | 否 | 脚本返回的结构化结果,JSON 序列化后写入 `runs.result_data_json`;建议 < 4 KiB |

**约定**
- 函数正常 return 一个 `RunResult` → 按 success 字段决定 `success` / `failure` 状态
- 函数抛异常 → 捕获后状态记为 `error`,异常 traceback 写入 stderr,`result_message` = `f"{type(e).__name__}: {e}"`
- 子进程 timeout(SIGTERM 后 5s SIGKILL) → 状态 `timeout`,`exit_code = -15`
- 子进程异常退出 (非 0) → 状态 `error`

### 3.4 脚本与主程序的解耦运行方案

**推荐方案:独立子进程 + JSON over stdio**

主程序与脚本通过 `subprocess.Popen` 启动:
- 命令:`python -u -m sandbox_runner --script-dir <abs> --run-id <n>`
- 主程序通过 stdin 写一条 JSON(包含 config + context 序列化形态),子进程从 stdin 读取
- 子进程导入脚本的 `main.run`,执行,把 `RunResult` 序列化为 JSON 写入 stdout 的**最后一行**(以特殊前缀 `__RUN_RESULT__` 标记)
- 普通 `print()` / logger 输出走 stdout/stderr,主程序实时收集并广播给 SSE
- `sandbox_runner` 是一个内置启动器(`backend/runner/sandbox.py`),不是用户编写的脚本

#### 3.4.1 推荐理由

| 维度 | 子进程方案(推荐) | 直接 import 方案 |
|---|---|---|
| 隔离性 | ✓ 崩溃不影响主进程 | ✗ 脚本 segfault 整个面板挂掉 |
| 依赖冲突 | ✓ 可用不同 venv / requirements | ✗ 全部脚本依赖共享一个进程 |
| 超时强杀 | ✓ SIGTERM/SIGKILL 干净利落 | ✗ Python 线程无法强杀,只能"礼貌"终止 |
| 实时日志 | ✓ stdout/stderr 天然独立流 | △ 需自定义 handler 转发 |
| 内存释放 | ✓ 进程结束 100% 回收 | ✗ 长期可能泄露 |
| 启动开销 | ✗ 100-300 ms cold start | ✓ 几乎为 0 |
| 调试 | △ 需要 attach debug 麻烦 | ✓ 直接断点 |
| 实现复杂度 | △ 中等(stdio 协议) | ✓ 简单 |

签到脚本一天跑几十次、单次几秒到几十秒,启动开销可忽略,**安全性和隔离性收益远大于成本**。

#### 3.4.2 stdio 协议

主程序 → 子进程(stdin 单条 JSON,以 `\n` 结束):
```json
{
  "config": { "...": "..." },
  "context": {
    "run_id": 123,
    "instance_id": 5,
    "instance_name": "B站-主号",
    "script_slug": "bilibili-daily",
    "script_dir": "/app/scripts/bilibili-daily",
    "data_dir": "/app/data/scripts/bilibili-daily/5",
    "timeout_sec": 300,
    "trigger_type": "scheduled",
    "attempt": 1
  }
}
```

子进程 → 主程序(stdout 普通日志 + 最后一行结果):
```
[2026-05-15 08:00:01] INFO 开始登录 B 站...
[2026-05-15 08:00:03] INFO 投币完成 (1/1)
__RUN_RESULT__{"success":true,"message":"签到成功,获得经验 +15","data":{"exp":15,"coins_used":1}}
```

子进程 → 主程序(异常):
```
[2026-05-15 08:00:01] ERROR 网络超时
__RUN_RESULT__{"success":false,"message":"NetworkError: timeout","data":{}}
```

若 stdout 没有 `__RUN_RESULT__` 标记行,主程序按"协议异常,状态记为 error"处理。

## 4. 调度引擎方案

### 4.1 进程拓扑选型

**方案 A**:APScheduler 与 FastAPI 同进程(`AsyncIOScheduler`)
**方案 B**:独立 worker 进程(`BackgroundScheduler` 或单独 Python 进程)+ 通过 DB/queue 与 API 通信

**推荐方案 A**,理由:
1. 个人/小团队场景,负载极低(几十 instance,每天百来次执行)
2. SQLite 单文件 + WAL 已足够支撑;独立 worker 反而引入分布式调度复杂度
3. 同进程便于直接复用 SQLAlchemy session、内存日志 broker(SSE 共享)
4. APScheduler 的 `AsyncIOScheduler` 与 FastAPI 的 `asyncio` event loop 天然集成
5. 真正运行脚本是子进程,调度器只是"按时启动子进程",CPU 不密集

**注意事项**(同进程方案的折衷):
- FastAPI worker 数 = **1**(uvicorn `--workers 1`),否则多进程同时跑调度会重复触发
- 高可用场景另议;v1 单机部署不考虑

### 4.2 JobStore 选型

**推荐:`MemoryJobStore`**(默认),不用 SQLAlchemyJobStore。理由:
- "调度任务"等同于"启用的 instance",事实源是 `instances` 表,JobStore 是冗余
- 启动时全量扫描 `instances`(WHERE enabled=true AND paused_until IS NULL OR paused_until < now),逐个 `scheduler.add_job`
- 每次 instance CRUD/启用/禁用/暂停 同步调用调度器接口
- 进程重启时全量重建,简单可控,无 SQLAlchemyJobStore 的 SQLite 锁问题

**若坚持用 SQLAlchemyJobStore 的注意事项**(备查):
- 必须开启 SQLite WAL 模式(`PRAGMA journal_mode=WAL`)
- `SQLAlchemyJobStore` 写 jobs 表与业务表共用同一 connection,长事务需谨慎
- 多线程触发同一 job 必须用 `coalesce=True` 避免堆积
- 错过执行用 `misfire_grace_time` 控制,默认 1 秒太短,建议 60 秒

### 4.3 任务注册/取消时机

| 触发时机 | 调度器动作 |
|---|---|
| 应用启动 | 扫描所有 `enabled=true AND paused_until IS NULL` 的 instance,逐个 `add_job` |
| 创建 instance | 若 `enabled=true`,立即 `add_job` |
| 更新 instance(cron/timeout/enabled 变更) | `reschedule_job` 或 `add_job/remove_job` |
| 启用 instance | `add_job` |
| 禁用 instance | `remove_job`(若有) |
| 暂停 instance | `pause_job`,并起一个一次性"恢复任务"在 `paused_until` 时 `resume_job` |
| 删除 instance | `remove_job`(若有) |
| 删除 script | 级联,所有其 instance 的 job 都 remove |
| 全局禁用 script | 所有其 instance 的 job 都 pause(不删,以便快速恢复) |
| 全局启用 script | 反之 |

`job_id` 命名:`instance:{instance_id}` 全局唯一。

### 4.4 任务执行流程

```
触发(scheduler 或手动 API)
   ↓
SchedulerService.execute(instance_id, trigger_type)
   ↓
[1] 获取并发槽位(asyncio.Semaphore,容量 = settings.concurrent_runs_max)
       └── 满则进入排队队列(简单 list,FIFO)
   ↓
[2] 加载 instance + 关联 script;判断状态(enabled/paused);若不应执行,记日志并返回
   ↓
[3] 创建 runs 行(status=pending, started_at=now, exit_code=null)
   ↓
[4] 解密 config_blob → dict;按 fields_schema 二次校验(防字段定义变化导致脏数据)
   ↓
[5] 准备 RunContext(创建 data_dir 若不存在);组装 stdio JSON
   ↓
[6] subprocess.Popen 启动 sandbox_runner;
       cwd = scripts/<slug>/
       env = 白名单变量 + RUN_ID/INSTANCE_ID
       stdin = pipe (写入 JSON 后关闭)
       stdout/stderr = pipe (异步读取)
   ↓
[7] runs.status = running;广播 status 事件给 SSE
   ↓
[8] 异步两路读取 stdout/stderr:
       ├── 实时 push 到 broker(SSE 转发)
       ├── 累计到内存 buffer(超 max_log_bytes 截断)
   ↓
[9] 等待子进程退出 OR 超时(asyncio.wait_for(proc.wait(), timeout))
       ├── 超时:proc.terminate(),5 秒后 proc.kill();status=timeout
       ├── 正常退出:解析 stdout 最后一行 __RUN_RESULT__;status=success/failure
       ├── 异常退出码:status=error
   ↓
[10] 写回 runs:finished_at, duration_ms, exit_code, status, result_message,
       result_data_json, stdout, stderr, *_truncated
   ↓
[11] 同步更新 instances 冗余字段:last_run_*, total_runs+=1, total_successes+=(1 if success)
   ↓
[12] 触发通知(NotificationService.dispatch(run))
   ↓
[13] 若失败且 attempt < max_retries:
       └── 调度一次性 retry job 在 retry_interval_sec * 2^(attempt-1) 秒后,trigger_type=retry, parent_run_id=当前
   ↓
[14] 释放并发槽位;触发排队队列下一个
```

### 4.5 失败重试

- `max_retries`:0 = 不重试,默认 0
- `retry_interval_sec`:首次重试等待秒数,默认 60
- 退避公式:第 n 次重试等待 = `retry_interval_sec * 2^(n-1)`,封顶 1 小时
- retry 触发的 run 行 `trigger_type='retry'`、`parent_run_id` 指向上次失败 run
- 重试链上限 = `max_retries`,达到后不再重试;最终通知按最后一次结果触发(避免每次重试都发通知刷屏)
- 用户手动取消 (`/runs/{id}/cancel`) 不触发重试

### 4.6 时区处理

- 数据库统一存 UTC(SQLAlchemy `DateTime(timezone=True)`,SQLite 实际存 ISO 字符串)
- `settings.timezone`(默认 `Asia/Shanghai`)是**用户时区**
- cron 表达式按用户时区解释:`CronTrigger.from_crontab(expr, timezone=ZoneInfo(settings.timezone))`
- API 输入时间格式:ISO 8601 带时区(如 `2026-05-15T08:00:00+08:00`),无时区视为用户时区
- API 输出时间格式:ISO 8601 UTC(如 `2026-05-15T00:00:00Z`),前端按用户时区渲染
- 修改 `settings.timezone` 后必须重新加载所有 cron job(`reschedule_all_jobs`)

## 5. 安全与加密

### 5.1 Fernet key 管理

**位置**:`data/encryption.key`(`data/` 是持久卷,Docker volume 挂载)

**生成时机**:
1. 应用启动时检查文件是否存在
2. 不存在 → `cryptography.fernet.Fernet.generate_key()` 生成 32 字节 base64 字符串
3. 写入 `data/encryption.key`,文件权限 `0600`(仅 owner 可读写)
4. 在应用日志中**强警告**:"已生成新加密密钥,务必备份此文件,丢失将无法解密所有 secret"
5. 启动时同时在 UI 顶部贴一个"密钥未确认备份"红色横幅,直到用户在"设置→安全"页点击"我已备份"按钮(写入 settings)

**Key Rotation**(v1 不实现,设计预留):
- `instances.config_version` 字段标识当时使用的 key 版本
- `data/encryption.key` 改为 `data/encryption-keys.json` 存多代 key + 当前主 key id
- 加密用主 key,解密用 config_version 对应 key
- 后台任务渐进重加密

**禁止**:
- 不允许通过 API/UI 读取 key 内容
- 不允许 key 出现在任何日志、报错栈、备份导出元数据(导出时若包含 key,需用户显式勾选并接受警告)

### 5.2 敏感字段加密

**加密粒度**:整个 `config_blob` 一次加密(不分字段),理由:
- 简单可靠,Fernet 调用一次
- 即便有"非 secret"字段也加密,降低误读风险
- 反正都要解密整个 blob 才能 spawn 脚本

**流程**:
- 创建/更新 instance:校验 config dict → `json.dumps(sort_keys=True).encode()` → `fernet.encrypt()` → base64 存 `config_blob`
- 调度执行:取 `config_blob` → `fernet.decrypt()` → `json.loads()` → 注入子进程
- API 响应:解密 → 按 fields_schema 把 secret 字段替换为 `null`,同时返回 `_secret_set` 字典

**API 自动脱敏规则**:
- secret 字段 GET 返回值固定为 `null`
- 同时返回 `"_secret_set": { "field_key": true|false }` 让前端区分"未配置"和"已配置但隐藏"
- PATCH 时:secret 字段未提交 = 保留原值;提交 `null` = 清空;提交字符串 = 更新
- `notification_channels.apprise_url` 同理:GET 返回 `<scheme>://***/***`(scheme 暴露便于识别渠道类型)

### 5.3 鉴权方式选型

**推荐:Server-side session + HttpOnly Cookie**,而非 JWT。理由:

| 维度 | Session Cookie(推荐) | JWT |
|---|---|---|
| 撤销登录 | ✓ 服务端删 session 即时失效 | ✗ 必须维护黑名单或短 TTL |
| XSS 防御 | ✓ HttpOnly,JS 拿不到 | ✗ 通常存 localStorage,XSS 直接拿走 |
| 实现复杂度 | ✓ 简单 | ✗ 需处理 refresh token、过期等 |
| 适用场景 | ✓ 单一 Web 前端 | △ 多端/SPA + 移动 |
| 无状态 | ✗ 服务端要存 | ✓ 完全无状态 |

本项目场景:Web 单端、单实例、不分布式 → session cookie 完胜。

**Session 实现**:
- 表 `sessions(id, user_id, token, created_at, expires_at, last_used_at, ip, user_agent)`,`token` = `secrets.token_urlsafe(48)`
- Cookie 名 `sid`,属性 `HttpOnly`、`Secure`(生产)、`SameSite=Lax`、`Path=/`、`Max-Age=settings.session_ttl_hours * 3600`
- 每次请求中间件查 session;过期/不存在 → 401
- 登出 = 服务端删行 + 客户端清 cookie

(为简化设计,sessions 也可纯内存 + Redis;v1 单机直接落 SQLite 表)

### 5.4 CSRF / CORS

**CSRF**:
- 因为用 SameSite=Lax cookie,GET 请求自动免疫,POST/PUT/PATCH/DELETE 来自跨站不会带 cookie
- 额外加一道:所有非 GET 请求要求自定义请求头 `X-Requested-With: fetch`,无此头返回 403(浏览器 form 提交无法添加自定义头)
- v1 不实现 CSRF token(场景简单,SameSite + 自定义头双保险足够)

**CORS**:
- 默认禁用 CORS;生产环境前后端同源(同一容器,nginx 路由)
- 开发环境通过 `settings` 中 `dev_cors_origins`(白名单)开启
- 不允许 `Access-Control-Allow-Origin: *` + `Allow-Credentials: true` 组合

### 5.5 子进程脚本运行的安全考量

**文件系统**:
- `cwd` 设为脚本目录 `scripts/<slug>/`,脚本可读自身资源
- 脚本应只写 `RunContext.data_dir` (`data/scripts/<slug>/<instance_id>/`),主程序保证目录存在并允许写
- v1 不强制 chroot/沙箱(单用户自用,信任脚本作者),但在文档中明确"data_dir 是唯一推荐写入位置"
- 容器整体以非 root 用户(uid 1000)运行,最小化逃逸危害

**环境变量**:
- 子进程默认**不继承**主进程环境变量(`env={}` 显式给一份新的)
- 注入的最小集合:`PATH`、`HOME`、`LANG=C.UTF-8`、`PYTHONUNBUFFERED=1`、`PYTHONIOENCODING=utf-8`
- `RUN_ID`、`INSTANCE_ID`、`SCRIPT_SLUG`、`DATA_DIR` 注入便于脚本读
- manifest `runtime.env_passthrough` 中声明的变量从主进程透传(典型用例:代理)
- 严禁透传 `ENCRYPTION_KEY`、`DATABASE_URL` 等主程序敏感配置

**网络**:
- v1 不限制(签到脚本本身就是发出站请求)
- Docker 容器层面用户可自行配置网络策略

**资源限制**:
- 超时由主程序计时强杀(SIGTERM → 5s → SIGKILL)
- v1 不做 CPU/内存 cgroup 限制(可在 docker-compose 整体限制)

## 8. 错误处理与日志

### 8.1 异常体系

```
AppException(Exception)                    # 基类,带 code / status_code / message / details
├── AuthError(401)
│   ├── InvalidCredentials
│   ├── SessionExpired
│   └── AccountLocked
├── PermissionError(403)
├── NotFoundError(404)
│   ├── ScriptNotFound
│   ├── InstanceNotFound
│   ├── RunNotFound
│   ├── ChannelNotFound
│   └── RuleNotFound
├── ValidationError(422)                   # 业务校验失败(区别于 Pydantic 的 422)
│   ├── ConfigSchemaError                  # 实例 config 不符合 fields_schema
│   ├── CronExprInvalidError
│   └── ManifestInvalidError
├── ConflictError(409)
│   ├── DuplicateName
│   └── ConcurrentRunConflict              # 例如 test 与 run 互斥
├── ResourceLimitError(429)                # 并发数/订阅数超限
├── ExternalServiceError(502)              # apprise 发送失败
└── InternalError(500)                     # 兜底,带 traceback id
```

每个异常带:
- `code: str`(机器可读,如 `INSTANCE_NOT_FOUND`)
- `status_code: int`(HTTP 状态)
- `message: str`(人可读,中文)
- `details: dict`(可选,字段校验错误等)

### 8.2 API 错误响应格式

```json
{
  "error": {
    "code": "INSTANCE_NOT_FOUND",
    "message": "未找到 ID 为 5 的实例",
    "details": {
      "instance_id": 5
    },
    "trace_id": "01HZX7..."
  }
}
```

- 所有错误统一走 `middleware/error_handler.py` 的 `AppException` 处理器
- Pydantic ValidationError 转换为 `code=VALIDATION_ERROR`,details 里放字段错误数组
- 未捕获异常 → `code=INTERNAL_ERROR`,生产环境不暴露堆栈,只返回 trace_id;trace_id 同时写应用日志

### 8.3 应用日志(推荐 loguru)

**理由**:
- 配置一行(`logger.add(...)`),自动彩色 + 格式化
- 文件轮转、压缩、保留策略开箱即用(`rotation="20 MB", retention="30 days", compression="zip"`)
- `logger.bind(...)` 支持上下文绑定(request_id、user_id、run_id)
- 与 stdlib logging 互通(`InterceptHandler` 转发 SQLAlchemy / uvicorn 日志)

**配置策略**:
- 输出目标:
  - `sys.stderr`(docker logs 收得到,LEVEL=INFO,生产环境)
  - `data/logs/app.log`(LEVEL=DEBUG,文件轮转)
  - `data/logs/error.log`(LEVEL=ERROR,只记错误便于排查)
- 格式:`{time:YYYY-MM-DD HH:mm:ss.SSS} | {level: <8} | {extra[trace_id]} | {name}:{function}:{line} | {message}`
- 上下文绑定:每个请求中间件注入 `trace_id`(UUID7)与 `user_id`,日志自动带

### 8.4 应用日志 vs 脚本执行日志的区分

| 类别 | 来源 | 落点 | 查看方式 |
|---|---|---|---|
| **应用日志** | FastAPI 路由、调度器、service 等 | `data/logs/app.log` + stderr | `docker logs` / 直接看文件 |
| **脚本执行日志** | 子进程 stdout/stderr | `runs.stdout` / `runs.stderr` 字段 + 实时 broker | UI 详情页 / SSE 实时流 |
| **审计日志**(可选 v2) | 用户登录、关键 CRUD | 单独表 `audit_logs` | 设置页面 |

脚本执行日志**不进入** loguru 流,仅广播 + 落 runs 表。理由:
- 脚本日志是业务数据,与运维日志诉求不同
- 避免脚本输出污染应用日志,影响 docker logs 信噪比
- runs 表带索引,按 instance/时间检索快;loguru 文件需要 grep

应用层会记录脚本执行的"元日志":
```
INFO  | run_id=123 | 启动子进程 instance=5 script=bilibili-daily timeout=300s
INFO  | run_id=123 | 子进程退出 exit_code=0 duration=2341ms status=success
WARN  | run_id=124 | 子进程超时,发送 SIGTERM
ERROR | run_id=125 | 子进程协议异常,未找到 __RUN_RESULT__ 标记
```

## 9. 附录

### 9.1 数据库初始化与迁移

- 使用 Alembic 管理 schema 变更
- 首次启动:`alembic upgrade head` 自动执行
- SQLite 连接事件钩子(`db/pragma.py`)在每个新连接上执行:
  ```
  PRAGMA journal_mode = WAL;
  PRAGMA synchronous = NORMAL;
  PRAGMA foreign_keys = ON;
  PRAGMA busy_timeout = 5000;
  PRAGMA cache_size = -20000;
  ```

### 9.2 启动顺序(`app/main.py` lifespan)

```
[startup]
1. 初始化日志(loguru)
2. 加载 settings 缓存
3. 初始化 DB engine + 应用 PRAGMA
4. 运行 alembic upgrade head(可选,docker entrypoint 也可承担)
5. 加载/生成 Fernet key
6. 启动 NotificationService(初始化 apprise 实例池)
7. 启动 SchedulerService:
   a. 注册内置周期任务(cleanup_runs / scan_scripts / housekeeping / resume_paused)
   b. 若 settings.script_scan_on_startup,执行一次同步扫描
   c. 加载所有 enabled instance 注册到调度器
   d. scheduler.start()
8. 挂载路由

[shutdown]
1. scheduler.shutdown(wait=True)            # 等待运行中的任务完成
2. 取消所有 SSE 连接,广播 end 事件
3. 关闭 DB engine
4. 关闭日志 sink
```

### 9.3 备份与恢复设计

**导出**(`POST /settings/backup/export`)
- 临时目录打包:
  - `db.sqlite3`(先 `.backup` 命令做一致性快照,避免 WAL 不一致)
  - `meta.json`:`{ version, exported_at, includes_key: bool }`
  - 可选 `encryption.key`(用户在请求中显式 `?include_key=true` 才包含,UI 上有红色警告)
- zip 后流式返回,文件名 `checkin-panel-backup-YYYYMMDD-HHMMSS.zip`
- **不**包含 `data/scripts/`(脚本运行时数据,通常不必备份)和 `data/logs/`

**导入**(`POST /settings/backup/import`)
- 上传 zip,先解析 `meta.json` 校验版本兼容
- 进入"维护模式":scheduler.pause()、阻塞新请求(返回 503 + Retry-After)
- 备份当前 `db.sqlite3` 和 `encryption.key` 到 `data/backups/before-restore-{ts}/`
- 替换文件
- 触发应用重启(返回 202 + 提示),由 docker `restart: unless-stopped` 拉起

### 9.4 通知模板默认值

默认模板(Jinja2 语法,支持 `instance`、`script`、`run`、`event` 上下文):

**title 默认**:
```
[{{ event | upper }}] {{ instance.name }} - {{ script.name }}
```

**body 默认**:
```
脚本: {{ script.name }} ({{ script.slug }})
实例: {{ instance.name }}
状态: {{ run.status }}
开始: {{ run.started_at }}
耗时: {{ run.duration_ms }} ms
{% if run.result_message %}
结果: {{ run.result_message }}
{% endif %}
{% if event in ['failure', 'error', 'timeout'] and run.stderr %}
--- stderr (尾部) ---
{{ run.stderr | tail(20) }}
{% endif %}
```

预置过滤器:
- `tail(n)`:取尾部 n 行
- `human_duration`:把 ms 转为 "1m 23s"
- `local_time(tz)`:转换到指定时区显示

### 9.5 性能预估与扩展点

- 单机 SQLite + WAL + 同进程调度,可稳定支撑:
  - 实例数 ≤ 1000
  - 单日 runs ≤ 100,000(stdout/stderr 各 < 256 KB)
  - 并发执行 ≤ 16(受单机 CPU 与脚本 I/O 限制)
- 突破上限时的演进路径(预留,v1 不实现):
  - DB:SQLite → PostgreSQL
  - 调度:同进程 → 独立 worker(`apscheduler` + `redis` lock 或 `celery`)
  - 日志:`runs.stdout/stderr` Text → 对象存储(S3/MinIO)+ 仅存 URL
  - 执行:子进程 → 独立 docker container(给每个 run 起一次性容器)

## 10. 总结要点(给后续编码 agent 的提醒)

1. **加密优先**:`Instance.config_blob` 是设计核心,任何路径上**永远不要**让明文 secret 进入数据库或日志
2. **冗余字段记得同步**:`instances.last_run_*` `total_*` 必须在 run 完成时同步更新,否则列表页数据失真
3. **APScheduler 单 worker**:uvicorn `--workers 1`,任何文档/示例都不要写 `--workers 4`
4. **子进程协议严格**:`__RUN_RESULT__` 前缀是契约,缺失即 error;sandbox runner 是唯一启动器
5. **时区一致**:DB 存 UTC,cron 解析用 `settings.timezone`,API 输出 UTC,前端转换
6. **secret 字段 PATCH 语义**:未提交 = 保留,这是用户体验关键,不要写成"未提交 = 清空"
7. **删除是软还是硬**:`scripts` DELETE 是 DB 删 + 不删磁盘;`instances` DELETE 是真删 + 级联 runs;**先在前端弹二次确认**
