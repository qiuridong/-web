## 3 个 hotfix 详细

### #1 executor.py PYTHONPATH 安全透传

**root cause** — `backend/app/scheduler/executor.py` `_build_env()` audit Critical #1 修复时一刀切禁 PYTHONPATH 透传(`_FORBIDDEN = {"PYTHONPATH", ...}`),假设第三方包在 `backend/.venv/Lib/site-packages`(本机 OK)。生产 Dockerfile 走 `uv pip install --target /deps` + `ENV PYTHONPATH=/deps`,子进程不继承 PYTHONPATH → import httpx 失败。

**修法** — 新加 PYTHONPATH 安全透传:
```python
parent_pythonpath = os.environ.get("PYTHONPATH", "")
if parent_pythonpath:
    safe_paths = []
    for p in parent_pythonpath.split(os.pathsep):
        abs_p = Path(p).resolve()
        # 拒绝 backend/ 本身 + 任何子路径(防 import app.*)
        if abs_p == _BACKEND_DIR or abs_p.is_relative_to(_BACKEND_DIR):
            continue
        safe_paths.append(p)
    if safe_paths:
        env["PYTHONPATH"] = os.pathsep.join(safe_paths)
```

**部署** — 先 hot patch(`docker cp executor.py + restart`)立刻恢复生产 cron;本次 MVP-5 完整 `docker compose build` 永久带上。

**教训** — audit fix 时 PM 应该 verified 生产部署方式(Dockerfile),不能只测本机。

### #2 ptfans `_check_logged_in` 正则误判

**root cause** — `scripts/ptfans/main.py` 的 `RE_USERNAME` 正则硬编码 `class="User_Name"`,基于 agent 当初 HAR 抓包的 `attendance.php` 样本。但 `/index.php` 当前 NexusPHP 主题(BambooGreen)用户区 class 名不同 → `_check_logged_in` 在已登录页面被判未登录 → raise NotLoggedInError → 误报 "cookie 已过期"。

**关键诊断手法** — PM SSH 容器内 `httpx.get(https://ptfans.cc/index.php, headers={Cookie: 用户 cookie})` 直接验证 server 响应:
- 旧失效 cookie:`HTTP 302 → Location: login.php` → server 真的不认
- 新刚登录 cookie:`HTTP 200 + 含 userdetails.php?id=21550` → server 真认但脚本判定失败 → 锁定脚本 bug

**修法** — 新增宽松 `RE_LOGGED_IN_HINT`,只要含 `userdetails.php?id=<数字>` 或 `logout.php` 链接即视为登录态;`_check_logged_in` 改用宽松版;`RE_USERNAME` 保留(用于 `_parse_user_info` 抽用户名,抽不到 fallback "unknown")。

**部署** — `scripts/` 是 host volume(`./scripts:/app/scripts`),scp 即生效,无需 restart。

**教训** — 严格抽取与登录态判定应解耦(严格抽取失败可降级为 unknown,登录与否判定应用宽松信号)。agent 自己报告里也提到"主题升级导致正则失效"风险,但没意识到不同页面用的可能是不同主题。

### #3 format.ts 时区识别(中国用户全部时间显示偏 -8h)

**root cause** — `frontend/src/lib/format.ts` 第 22-23 行 `parseISO(input)` 解析后端返回的 naive UTC 字符串(SQLAlchemy + `datetime.utcnow()` 产出,无 Z 后缀)时 **当作本地时间**。中国用户(UTC+8)看到的所有 timestamp 偏 -8 小时。

**用户表现** — 报告 "coklw 1 点 cron 没触发",其实成功了(Run #13 status=success "今日已签到",started_at UTC 2026-05-18 17:00 = CST 2026-05-19 01:00),但 UI 把这个时间显示成 "2026-05-18 17:00" 让用户以为是昨天下午。

**修法** — `toDate()` 函数:string 输入若无显式时区后缀(`Z` / `+HH:MM` / `-HH:MM`),自动补 `Z` 假定 UTC:
```typescript
const hasTz = /([Zz]|[+-]\d{2}:?\d{2})$/.test(input);
return parseISO(hasTz ? input : input + 'Z');
```

**影响范围** — 所有 timestamps 显示的地方:`/runs` 列表 / `/dashboard` 即将运行 / 时间线 / 实例 last_run_at / "X 小时前" 相对时间 / Run 详情 started_at/finished_at 等。

**教训** — 后端 datetime 序列化应显式带 Z 后缀(`datetime.now(timezone.utc).isoformat()` 或 schema 加 serializer),前端"假定 UTC"是 fallback,根治是后端契约清晰化(下次 audit 时可加 issue)。

## 真实生产数据(已 verified — 含 1 个"假成功"业务陷阱)

| Run | 触发 | UTC | CST | 状态 | 真实业务含义 |
|---|---|---|---|---|---|
| coklw #15 | manual | 2026-05-19 02:10 | 2026-05-19 10:10 | ✅ **真签** | "您获取了签到奖励。您是第 2813 个签到人士。" |
| coklw #13 | scheduled `0 1 * * *` | 2026-05-18 17:00 | 2026-05-19 01:00 | ⚠️ **假成功** | "今日已签到(状态接口确认)" — **server 翻页未到,这个"今日"还是 5-18 的签到记录在生效**,脚本 skip_if_signed=true 看到"已签"就跳过 → 没真签到 5-19 这天 |
| ptfans #14 | scheduled | 2026-05-19 01:00 | 2026-05-19 09:00 | ✅ success | PTFans cron 9:00 触发 |

### 业务陷阱:跨日临界点的 skip_if_signed

WordPress / NexusPHP PT 站"今日"切换时刻不一定是 00:00,可能是 04/06/08/09 点(server 时区 + 业务逻辑决定)。如果 cron 触发时 server 还没切到新的一天:
- 状态接口返回"今日已签"(实际是昨天的签到记录)
- 脚本 `skip_if_signed=true` 看到"已签"就 skip → **静默跳过真签到机会**
- run.status=success(无报错),但实际没签

**用户观察 + 修法**:用户已把 coklw cron 从 `0 1 * * *` 改回 `0 8 * * *`(接近 manifest default `0 9 * * *`)。**明天 5-20 早上 8 点 cron 触发,Run #16 的 message 见分晓**:
- 若 "您获取了签到奖励..." → 8 点已够,cron 配置合理
- 若 "今日已签到" → 8 点仍太早,需要再推到 9 / 10 点

### 教训(coklw cron 这件事)

- ❌ PM 不能只看 `run.status=success` 就报"成功",必须看 message 语义(尤其 "skipped" 类语义)
- ❌ `skip_if_signed=true` + 跨日临界 cron 的组合有静默漏签风险
- ✅ default cron `0 9 * * *` 是有道理的(避开 server 翻页临界),用户不该改太早
- ✅ 未来可考虑:manifest 加 `min_server_hour` 字段 / 脚本加"判断 server 翻页"逻辑 / 或者干脆 cron 默认就在 server 已经翻页之后的稳妥时刻
