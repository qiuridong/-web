## 1. TL;DR

## 2. 设计思路 / 关键决策

### 2.1 Pull vs Push

| 模型 | 优点 | 缺点 |
|---|---|---|
| **Pull**(选定) | agent 完全被动,主面板不需要追踪 agent 在线;版本错位自纠 | 第一次跑实例稍慢(多一次 HTTP 拉取) |
| Push | 上传完立即同步,UX 反馈强 | 需要主面板 push agent / WebSocket / sync_script task 类型 / DB migration |

选 Pull:**实现成本 ~1/3**,稳定性 / 容错性 / 向后兼容性都更好。

### 2.2 按需触发 vs 周期 polling

- **按需**(选定):agent 拿到真实 run task 时,先 `_ensure_script_synced` 比对 sha256 → 不一致拉。
- 周期 polling:每 5 min 调一次 `/agent/scripts/required` 检查应该有哪些 slug。

选按需:节省 HTTP 调用,实例不跑的脚本就不浪费带宽。新节点接入第一次跑实例时自然同步。

### 2.3 上传时"选节点" = UI 预提示

- 用户痛点(5-25 原话):"以及这个脚本上传到哪个 vps 节点的这些应该要有这个选项"
- 实现:UploadScriptDialog 加 Checkbox 多选 + 上传成功 toast,**但不调专门 push API**
- 实际同步发生在:用户创建/跑实例 → agent `_ensure_script_synced` 触发
- 优势:不需要主动 push 机制(避免 sync_script task 类型 / WebSocket / DB migration)

若用户反馈"我希望上传后立即看到节点同步成功",再升级到主动 push。

### 2.4 不引入 DB schema 改动

- 不加 `script_node_deployments` 表
- 不加 `runs.kind` 字段(没有 sync_script task)
- 优势:本次部署**无需 alembic upgrade**,极简

### 2.5 bundle 过滤策略

| 排除目录 | 原因 |
|---|---|
| `.backups/` | 在线编辑生成的旧版备份 |
| `__pycache__/` | Python 字节码缓存 |
| `data/` | 历史 / 用户运行时数据 |
| `.git/` | 仅在开发期出现 |
| `_dry_run_data/` | dry-run 产物 |
| `node_modules/` | 万一脚本含 nodejs 工具 |
| `.pytest_cache/` | 测试缓存 |

| 排除后缀 | 原因 |
|---|---|
| `.pyc / .pyo / .pyd` | Python 编译产物 |
| `.log / .tmp` | 运行时垃圾 |
| `.DS_Store` | macOS metadata |

点开头文件仅保留白名单:`.gitignore` / `.env.example`(其它 `.foo` 都跳)。

## 3. 实施细节

### 3.1 Backend `script_upload_service.compute_script_bundle()`

```python
def compute_script_bundle(scripts_root: Path, slug: str) -> tuple[bytes, str, int]:
    """打 scripts/<slug>/ 为 zip,返 (bytes, sha256_hex, size)。"""
    # 路径安全 + ScriptNotFound 校验
    # 遍历 rglob('*') 过滤 BUNDLE_EXCLUDE_DIRS / SUFFIXES
    # 写入 ZipFile(BytesIO, ZIP_DEFLATED, level=6)
    # 防大小爆炸:raw_total > MAX_ZIP_TOTAL_BYTES → PayloadTooLarge
    # 返回 (bytes, hashlib.sha256().hexdigest(), len(bytes))
```

### 3.2 Backend `agent.py` 新增 2 endpoint

```python
@router.get("/scripts/{slug}/manifest")
async def agent_get_script_manifest(slug, node, db):
    script = db.scalars(select(Script).where(Script.slug == slug)).first()
    if script is None: raise ScriptNotFound(...)
    _data, sha, size = script_upload_service.compute_script_bundle(...)
    return {"slug", "version", "manifest_hash", "bundle_sha256", "bundle_size"}

@router.get("/scripts/{slug}/bundle.zip")
async def agent_get_script_bundle(slug, node, db) -> Response:
    # 同样检查 + compute_bundle
    return Response(
        content=data,
        media_type="application/zip",
        headers={
            "Content-Disposition": f'attachment; filename="{slug}.zip"',
            "X-Bundle-SHA256": sha,
            "X-Bundle-Size": str(size),
            "Cache-Control": "no-store",
        },
    )
```

### 3.3 Agent `_ensure_script_synced()`

8 步流程:
1. GET `/api/v1/agent/scripts/<slug>/manifest`(timeout=15s)
2. 读本地 `scripts/<slug>/.bundle_sha256` marker
3. `local_sha == remote_sha and main.py exists` → return
4. GET `/api/v1/agent/scripts/<slug>/bundle.zip`(timeout=60s)
5. 客户端二次校验 `hashlib.sha256(bytes).hexdigest()`,不匹配 → RuntimeError
6. 解压到 `.tmp-{slug}-{ts}/`,**两层 Zip Slip 防御**:
   - 第一层:遍历 namelist 拒绝 `/` 开头 / `..` 段 / 单文件超限
   - 第二层:解压时 `target.resolve().relative_to(tmp_dir.resolve())` 兜底
7. 备份旧目录 `script_dir → .backup-{slug}-{ts}/` → `os.replace(tmp_dir, script_dir)` → 失败回滚
8. 成功 → 清备份目录

调用时机:`_execute()` 开头,**网络/拉取失败仅 log 不抛**,让 `main.py 不存在` 分支兜底报错(老 agent 升级前也能继续工作)。

## 4. 异常处理 / 容错矩阵

| 场景 | 行为 |
|---|---|
| 主面板没有此 slug(404) | `_ensure_script_synced` log warning + return,让 main.py 检查兜底 |
| 网络抖动拉 manifest 失败 | 同上 |
| bundle.zip sha256 不匹配 | **抛 RuntimeError**(系统级问题不该继续跑) |
| Zip Slip / .. / 绝对路径 | 抛 RuntimeError |
| 单文件 > 256 KiB | 抛 RuntimeError |
| 原子 replace 失败 | 自动回滚备份 → 抛 RuntimeError |
| backend 找不到 scripts/<slug>/(磁盘缺) | ScriptNotFound 404 |
| backend bundle 超 1 MiB | PayloadTooLarge 413 |
| backend db 没有 Script 记录 | ScriptNotFound 404 |
| backend 解析 manifest 失败 | (不影响 — bundle compute 不查 manifest) |

向后兼容:**老 agent 不升级,行为不变**(主面板新 endpoint 不被调用)。

| 项 | 触发条件 |

## 8. 教训 / 设计要点

- **客户端二次校验 sha256** 是必须的:HTTP 传输虽然 TLS 保护,但攻击者控制服务端 OR 中间代理污染时,sha256 是真终极防线。
- **两层 Zip Slip 防御** 不嫌多:namelist 遍历 + 解压时 resolve 校验,任何一层挂掉都不会落到磁盘。
- **MVP 不引入 DB 改动** 是巨大胜利:本次部署不用 alembic upgrade,docker rebuild 即可。
- **向后兼容**:`_ensure_script_synced` 网络失败仅 log,老 agent / 主面板没此脚本的情况都不影响现有流程。
- **UI 标记 vs 实际推送** 这种"半实现"在 MVP 阶段是合理的 — 用户能看到选项,实际同步在更自然的时机触发,既满足需求又不增加复杂度。
- **Bash classifier 短暂不可用**(~3 分钟)期间纯写代码不操作 git,恢复后顺利 commit + push。Auto mode 给了一个 "this is temporary" 的明确信号,等候比尝试绕过更明智。
