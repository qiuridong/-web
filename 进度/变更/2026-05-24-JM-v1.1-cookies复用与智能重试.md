# 2026-05-24 · JM v1.1.0 cookies 复用 + 智能重试

## 30 秒摘要

1. **5-24 上午 v1.0 host 首跑成功** ✅ — `JCoin:30 + EXP:100`,30 秒完成(09:48:46-09:49:16),证明 v1.0 selenium 路径 production 可用 + 5-23 sign 反爬空响应**已确认是临时偶发**(不是持续升级)
2. **用户提需求**:"签到失败要兜底二次签到,**使用第一次过 CF 的值**,不要反复过 CF;5 分钟间隔重试 3 次;cookies 真过期才重过 CF;3 次都失败 → 通知用户(web 项目已有,只需整合)"
3. **v1.0 痛点**:`run()` 重试循环里调 `_do_sign_in(...)`,后者内部 `_get_cookies_via_browser`,**每次重试都重新启 Chrome 过 CF**(60-120s × N 次 + IP 信任分 N 倍消耗)— 浪费严重
4. **v1.1 重构**:
   - 拆 `_do_sign_in` → `_do_login`(只 login 返 session)+ `_do_sign_only`(用现成 session 调 sign)
   - 新加 `JmCookieExpired` 异常类(server 返 `{"msg":""}` 触发)
   - `run()` 顶层管 cookies/session lifecycle,智能决策何时重过 CF(主动 ttl 检查 + 被动 cookies 失效检测)
   - manifest 加 `retry_interval_sec`(默认 300s = 5min)+ `cf_clearance_ttl_sec`(默认 1800s = 30min)
   - `default_timeout_sec` 4500 → 6000(留够 5min × 2 间隔)
5. **部署**:scp 到 host VPS-JM + 主面板 + chown + docker restart backend → SQLite scripts 表更新 jmcomic 版本 1.0.0 → **1.1.0**
6. **通知集成**:**不在 v1 代码里加 apprise**(干净分层) — 等 MVP-1 通后,远程 agent 派 jmcomic 任务到 VPS-JM 跑,结果回传主面板,主面板 `dispatch_run_event` 自动触发用户配置的通知规则(Telegram / 微信 / 邮件)

## 新行为流程(v1.1)

```
1. cron 触发 09:00:01 北京(host crontab 不变)
2. v1 内部 random_delay 0-3600s 抽数(避开整点 1 小时窗口)
3. _ensure_dependencies(已装,秒过)
4. refresh_cf_and_login() ← 初次,启 Chrome 60-120s 过 CF + login
5. 重试循环 attempt = 1..3:
   ├─ if (cookies_age > cf_ttl 1800s) → refresh_cf_and_login() ← 罕见
   ├─ try _do_sign_only(session) ← 只调 sign,**复用 session**
   │   ├─ 成功 → return + logout ✅
   │   ├─ JmAlreadySignedToday → return + logout ✅
   │   ├─ JmCookieExpired(empty msg)→ refresh_cf_and_login() + 立即下一轮
   │   ├─ JmSignEndpointError → sleep 300s + 下一轮(复用 session)
   │   └─ JmNetworkError → sleep 300s + 下一轮
   └─ 3 次全失败 → return failure + 完整 diag
```

## 经验教训

| 教训 | 说明 |
|---|---|
| **重试不应该浪费昂贵资源** | v1.0 每次重试启 Chrome 是反模式 — 应该把"昂贵资源"(CF cookies / login session)的生命周期管理上提到顶层 |
| **善用 closure 管 lifecycle** | `refresh_cf_and_login()` nonlocal closure 比 class 更轻量,适合单脚本场景 |
| **专属异常驱动状态机** | `JmCookieExpired` 一个新异常类,run() 的状态机就清晰了(被动检测 + 自动恢复) |
| **保守 TTL** | cf_clearance 真实有效期未知 → 取经验下限 30 min,避免静默失效(浪费一次重试) |
| **诚实分层** | 通知不应耦合到脚本代码 — 主面板 notification_service 是平台职责,脚本只返 RunResult |
| **及时改进 timeout 预算** | retry_interval 从 30 → 300 时,要同步加 default_timeout_sec(否则 cap 后用户没意识到) |
