## 1. 设计决策

### 复用 vs 新建

**最初设计**:新建 `app_settings` 表 + `AppSetting` model + 专属 appearance API + 文件上传 endpoint + nginx 静态服务,~5-6h。

**调研发现** `app/db/models/setting.py` + `settings_service.py` + `GET/PUT /api/v1/settings/{key}` **早已存在**(MVP-1 / 设计稿 § 1.8),通用 KV + 白名单 + 校验 + 缓存 + ensure_defaults seed 都齐全。

→ **删掉刚写的 0004 migration + AppSetting model + appearance API,改为复用**:
- `appearance` 加到 `DEFAULT_SETTINGS` 白名单
- value 是 dict(整个 appearance blob),`set_value` 自动 json.dumps 存 `value_json`
- 前端调通用 `GET/PUT /api/v1/settings/appearance`

**收益**:
- backend 从 4 个新文件 → **1 个文件改 80 行**
- 无 migration(无破坏 schema 风险)
- 无新 endpoint(无 API 表面积扩张)
- 后续加新 setting 类型继续走相同模式

### 图片存储:base64 内联 vs 文件系统

**选 base64 内联**:
- 单字段上限 3 MB(validator 强制)
- 存 `settings.value_json`(JSON 字符串)
- 前端 FileReader 转 base64 data URL → PUT 整个 dict
- 优点:**无 docker volume / 无 nginx serve / 无文件清理**,部署简单
- 缺点:每次 GET 传全部图(用 staleTime: Infinity 缓存,只第一次)

### Validator 设计

`_validate_appearance()` 宽松校验:
- 必须 dict
- 字符串字段长度上限(site_title 128 / sidebar_logo_text 8 / 图 data URL 3 MB)
- 数值字段范围(background_blur 0-40 / background_opacity 0-1)
- background_blend_mode 在 CSS 合法值集合
- **允许字段缺失**(用 hook 端 DEFAULT_APPEARANCE 兜底,向后兼容老 setting)

## 7. 教训

- **复用 > 新建**:写 4 个新文件之前 grep 一下,可能现有的就够。本次省 ~2.5h。
- **base64 + DB** 适合外观这种"小图 + 不频繁更新"场景。如果是用户上传 100 MB 视频之类,文件系统 + CDN 更合适。
- **AppLayout 这种 root 组件改 useAppearance,影响全局** — 多 PR 并行容易冲突,串行更稳但慢。本次接受冲突等用户 merge 顺序拍板。
