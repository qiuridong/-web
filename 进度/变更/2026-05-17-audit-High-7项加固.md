## 预期 Breaking Change

### ⚠️ 1 项预期 breaking(就是 audit #9 修复目标)

**生产 `/docs` 与 `/openapi.json` 升级后默认 404**。

- 原因:`Settings.expose_docs` 默认 `True` → 改为 `None` → `is_docs_exposed()` 在 `environment="production"` 时返 `False`
- `environment` 字段 default 已经是 `"production"`,所以**生产部署后立刻生效**
- 用户/运维想强制开启 → `.env` 加 `EXPOSE_DOCS=true`(`.env.example` 已补示例)
- 这是 #9 的修复价值核心(原生产 36 个端点 schema 无鉴权暴露给任意访客 = 攻击面枚举)

### ✅ 0 项实际 breaking(其它 6 项 + ADR 锁定)

- #7 N+1:同样的 JSON 响应,只是 SQL 从 N+1 条变 1 条(性能修复)
- #10 cron 预校验:之前能存非法 cron 然后调度炸,现在直接 422 拒绝,前端早就有同样校验,**生产无影响**
- #11 alias:`response_model_by_alias=True` 是显式声明,响应字段名仍是 `_secret_set`,前端无感知
- #12 503:把 `RuntimeError` → 500 改成 `ServiceUnavailable` → 503,前端如果只看 `res.ok`(`status` 200-299)无差;特殊状态码处理也能容(503 是更准确的语义)
- #13 order:前端 hooks 已声明 `'asc' \| 'desc'`,不会传非法值;新 pattern 校验只对手工调 API 的用户严格
- #14 ADR:**完全不改行为**,只加注释
- #15 backup/import 200→202:前端 `if (!res.ok)` 用 `res.ok`(200-299 都过),新增的 `staging_dir` 字段是**多出来**的(前端不消费也无影响)

## 没做的 / 跳过项

- **#15 真自动重启**(`os.execv` 后 docker compose 拉起新进程):风险评估为"高,无回滚",留给运维手动 `docker compose restart`;staging dir 让用户至少看到文件落到哪 + 知道要手动动作,HTTP 202 也是正确语义,设计稿 § 9.3 v1 trade-off 仍成立
- **#14 改默认 "id 小者优先"**(更稳定):会让用户已有规则配置的实际去向反转(若两条同 priority),得不偿失;仅锁 ADR 即可
