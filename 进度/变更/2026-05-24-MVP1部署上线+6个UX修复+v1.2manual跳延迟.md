## 6 个 UX bug fix 详解

### 1. heartbeat 500(SQLite writer lock)
**根因**:`authenticate_agent` 每次校验 token 后 `node.last_seen_at = now(); db.flush()` → 与 heartbeat 自己 `update_heartbeat` 的 UPDATE 撞 SQLite 单 writer lock(agent 端 30s heartbeat + 30s poll long polling 同时到达)
**Fix**:`authenticate_agent` 删 update + flush,只读校验;`last_seen_at` 由 `update_heartbeat` 专责(30s 一次,够用)
**文件**:`backend/app/services/node_service.py`

### 2. 节点卡片"再次查看安装信息"
**需求**:用户创建节点时 token 弹一次后关掉,如想查 install 命令模板没法看
**Fix**:NodeCard 加 Terminal 按钮(终端图标)→ 弹 TokenDisplayDialog `mode='replay'`(token placeholder `<YOUR_TOKEN_HERE>` + 说明文字 "token 已无法找回,如需新 token 请点重新生成")
**文件**:`frontend/src/pages/nodes/NodeList.tsx`

### 3. 节点按钮 Tooltip + 样式
**需求**:原生 title 提示难看,按钮无 hover 视觉反馈
**Fix**:4 个按钮(Terminal/Power/Refresh/Trash)全包 shadcn `<Tooltip>` + 加 hover bg 颜色(主色/警示/危险)
**文件**:同上

### 4. 创建实例报"未知错误"(可能根因)
**根因猜测**:backend `_serialize_instance_for_detail` 没返 `node_id` 字段 → frontend POST 200 后 React Query 把新 instance 入 cache,可能因 InstanceDetail schema 缺字段触发解析错误显示"未知错误"
**Fix**:
- `_serialize_instance_for_detail` 加 `"node_id": instance.node_id`
- `_serialize_instance_for_list` 加同上
- `InstanceListItem` Pydantic schema 加 `node_id: int | None = None`
- `InstanceDetail` Pydantic schema 加同上
- `frontend/src/api/hooks/instances.ts` `InstanceDetail` TS 加 `node_id?: number | null`
**文件**:`backend/app/services/instance_service.py` + `backend/app/schemas/instance.py` + `frontend/src/api/hooks/instances.ts`

### 5. 编辑实例节点显示默认 local
**根因**:backend `_serialize_instance_for_detail` 不返 node_id → frontend 拿 `instance.node_id` 是 undefined → fallback `?? 1`
**Fix**:#4 同步 fix 后,frontend 拿到真 node_id 显示对的;`InstanceFormSheet.tsx` 把 `(instance as unknown as { node_id?: number })?.node_id ?? 1` 改为 typed access `instance?.node_id ?? 1`
**文件**:`frontend/src/components/common/InstanceFormSheet.tsx`

### 6. "● 未知"误导
**根因**:新建实例 `last_run_status` 是 null,frontend `instanceStatusToBadge` default 返 'unknown' → StatusBadge 显示"未知"让用户以为出错
**Fix**:
- StatusBadge `Status` type 加 `'never_run'`
- STATUS_META 加 `never_run: { label: '待运行', dotColor: 'text-muted-foreground/70', pulse: false }`
- 2 个 `instanceStatusToBadge` fn(InstancesPanel + InstanceList)开头加 `if (s === null || s === undefined || s === '') return 'never_run'`
**文件**:`frontend/src/components/common/StatusBadge.tsx` + `frontend/src/components/common/InstancesPanel.tsx` + `frontend/src/pages/instances/InstanceList.tsx`

## v1.2 main.py: manual 跳 random_delay

**用户需求**:"立即运行应该是不受到随机事件的影响"
**Fix**:
```python
trigger_type = str(getattr(context, "trigger_type", "scheduled") or "scheduled")
if trigger_type == "manual" and delay > 0:
    logger.info(f"trigger_type=manual(用户立即触发)→ 跳过 random_delay({delay}s)立即开始")
    delay = 0
```
**逻辑**:
- web 上点 **立即运行** → trigger_type='manual' → **跳 random_delay,直接进 Chrome 阶段**
- cron 自动定时 / scheduler resume / agent 周期任务 → trigger_type='scheduled' → **走 0-3600s 随机延迟**(scheduled 错峰避风控合理)

**部署位置**:
- agent 节点 `/opt/signin-agent/scripts/jmcomic/main.py`(下次拉 task 自动用新版,**不用 restart agent**)

## 经验教训

| 教训 | 适用 |
|---|---|
| **SQLite 单 writer + 高并发请求** 必撞 lock — 中间件层不要做写操作,只做读校验,写操作集中在专门 endpoint(heartbeat) | 任何 FastAPI middleware 写 DB 的场景 |
| **Backend schema 加字段后,3 处都要补**:Pydantic schema + service serialize + frontend TS — 缺一就 react query 反序列失败显"未知错误" | 任何加 model 新字段的全栈改动 |
| **null vs unknown 状态语义要分清** — null 是"还没发生",unknown 是"发生了但识别不了",前端 UX 不应混淆 | 任何状态显示组件 |
| **trigger_type 是脚本/agent 应该感知的** — manual 不该走 random_delay,scheduled 走是合理的;脚本契约要在 context 透传这个 | 任何调度类脚本 |
| **MVP-1 e2e 真测的"成功"不是签到成功,是链路通** — 当业务因外部因素(IP 信任分)预期失败时,要看链路上每一步是否按设计走 | 任何 e2e 验证 |
| **Xvfb / Chrome driver crash 后进程残留** — 需要 finally 兜底 + 启动时 pkill 清理上一次的残留 | 任何 selenium 类脚本 |
