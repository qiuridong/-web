# 2026-05-26 · 模板 dry-run 短路 + 三层提示防开发者踩坑

> 5-26 上午用户实测 PR #1 上传链路时发现 422,根因是平台模板自身的 dry-run 设计漏洞。
> 不只修模板,从 3 个层级"防其他开发者也踩同样的坑"。

## TL;DR

- **现象**:用户硬刷拿新 hash + 下载模板 → 拖回上传 → 失败 `HTTP 422`,前端面板没显示 detail
- **诊断**:SSH 主面板 docker logs grep upload → 看到 `dry-run 失败 exit_code=1 timed_out=False`
- **根因**:`sandbox_runner.main()` 末尾 `return 0 if payload["success"] else 1` — `RunResult.success=False` 直接转 exit_code=1;模板 main.py 在空 config 下早 return `RunResult(success=False, "缺少 username...")` → exit_code=1 → backend 判 dry-run 失败 → 422
- **修复 3 层**:
  1. **模板 main.py** 加 dry-run 短路 + 重墨注释 + manifest description checklist
  2. **前端 ErrorPanel** 智能识别 dry-run failure,弹"最常见原因 + 解决方案 + 代码片段 + 跳指南"
  3. **开发指南 Sheet** 新增"🔍 dry-run 短路 · 上传前必读"专属章节
- 走 PR `hotfix/dry-run-guidance`(独立于 PR #4 推送同步)

## 1. 根因链路

```
模板 main.py 第一行 def run(config, context):
  ↓
config = {}(平台 dry-run 用空 config)
  ↓
username = config.get("username") or ""  # = ""
  ↓
if not username:
    return RunResult(success=False, message="缺少 username 或 password 配置")
  ↓
sandbox_runner._emit_result(payload)
  ↓
return 0 if payload["success"] else 1   # success=False → return 1
  ↓
子进程 exit_code = 1
  ↓
backend script_upload_service.dry_run_script():
  passed = (exit_code == 0 and not timed_out)  # False
  ↓
raise ValidationError("dry-run 失败 exit_code=1 timed_out=False")
  ↓
HTTP 422
```

这是 PR #1 模板设计的疏漏 — 模板"看起来对"(对真实跑没问题),但**永远不可能通过 dry-run**。

## 2. 3 层修复

### 2.1 模板 main.py + manifest.yaml(`frontend/src/lib/script-template.ts`)

**`MAIN_PY_TEMPLATE` 加 dry-run 短路段**(在 run() 第一行):

```python
# ============================================================
# ⚠️ dry-run 短路 — 不要删除这段!!
# ============================================================
# 平台上传脚本时,会用 **空 config + run_id=0 + instance_id=0** 调一次 run()
# 做"dry-run 自检",验证你的脚本能正常加载 + 执行 + 返 RunResult。
#
# sandbox_runner 协议:run() 返 RunResult.success=False → 子进程 exit_code=1
#                       → 平台判 dry-run 失败 → 上传被拒(HTTP 422)。
#
# 如果你删了这段,run() 一进来发现 config 缺 username/password
# 就 return RunResult(success=False, ...),上传**必失败**。
# ============================================================
if context.run_id == 0 and context.instance_id == 0:
    logger.info("dry-run 模式:跳过字段校验,返成功示范")
    return RunResult(
        success=True,
        message="dry-run OK · 模板已正常加载",
        data={"dry_run": True},
    )
```

**`MANIFEST_TEMPLATE` description 顶部加 checklist**:

```yaml
description: |
  ⚠️ 提交前 checklist:
  - main.py 顶部的 "dry-run 短路" 段不能删(否则上传必失败 HTTP 422)
  - slug 改成与目录名一致
  - 至少改一次 version
  ...
```

### 2.2 前端 ErrorPanel 智能提示(`UploadScriptDialog.tsx`)

422 失败时,如果 `detail.dry_run.passed === false || detail.dry_run.exit_code !== 0`,在错误面板**主动弹**"最常见原因"块:

- 黄色 warn 边框 + 📖 BookOpen 图标
- 标题:**"最常见原因:run() 在 dry-run 时返了 success=False"**
- 简短解释:平台用 0/0 跑空 config,sandbox_runner 把 success=False 转 exit_code=1
- 解决代码片段(可直接复制):
  ```python
  if context.run_id == 0 and context.instance_id == 0:
      return RunResult(success=True, message="dry-run OK")
  ```
- "📖 查看完整说明" 按钮 → 调 `onOpenGuide`(父组件传 `setGuideOpen(true)`)直接弹开发指南 Sheet
- 如果 `timed_out=true` 额外提示"脚本可能在 run() 里 sleep 太久,dry-run 默认 30 秒超时"

下面的"详细信息" pre 块标题动态:dry-run 时显示 **"脚本 stderr"**,其他错误显示 **"详细信息"**。

### 2.3 ScriptDevGuideSheet 新增 dry-run 章节

在 "🐍 main.py 协议" 之后、"📥 context 字段" 之前(显眼位置,讲完 run() 立刻讲 dry-run),加 **"🔍 dry-run 短路 · 上传前必读"** 章节:

- Callout warn:**"这是最常见的上传 422 失败原因"**
- 平台 dry-run 是什么(0/0 + 空 config)
- 为什么会 422(sandbox_runner 协议 success=False → exit_code=1)
- 正确写法(完整代码示例)
- Callout info:"下载平台模板的 main.py 已经写好这段,只要不删就 OK"

## 6. 教训

- **任何"看起来对"的模板代码,必须本地跑一遍完整上传流程验证** — 我 PR #1 写模板时只想着"真实跑的语义",没考虑 dry-run 这一特殊场景
- **错误提示要"教育"而不只是"展示错误"** — `dry-run 失败 exit_code=1` 这种诊断信息对人类无意义,得告诉用户"为什么 + 怎么修 + 链接到完整说明"
- **3 层防线**:
  - 模板自带正确代码 + 强警告注释(防止删除)
  - 文档(开发指南)有专章节(对照阅读)
  - 前端错误页智能提示(踩坑时主动提示)
- **PR 拆分纪律**:hotfix 跟 feat(PR #4)分开提,即使代码在同一个分支基础上,也要走独立 PR
