## 1. 根因

- backend `DELETE /api/v1/scripts/{slug}?confirm=true&delete_files=true` 早已存在(MVP-5 / `script_upload.py`)
- frontend hook `useScriptFullDelete()` 也已存在(`useScriptFiles.ts:263`),`onSuccess` toast `"脚本及其磁盘文件已删除"`
- **ScriptList.tsx 只用了 `useDeleteScript`**(软删,line 116),`useScriptFullDelete` 没 wired
- 用户在三点菜单点 "移除登记" → backend `DELETE ?confirm=true`(无 delete_files)→ 删 DB 行 → 下次 scan_scripts_dir 把磁盘 scripts/my-script/ 又 ADD 回 DB → 用户看到脚本"复活"

### 2.4 AlertDialog 双模式

按 `deleteTarget.mode` 切换:
- title:`"从登记中移除"` vs `"彻底删除"`
- description:软删强调"磁盘文件保留,下次扫描仍可重新发现(适用临时下架)" / 硬删强调"永久删除磁盘文件 `scripts/<slug>/`,**此操作不可恢复**"
- action button:`'确认移除'` vs `'彻底删除'`
- mutation:三元 `deleteTarget.mode === 'full' ? removeFull : remove`

## 5. 教训

- **UI 按钮存在 ≠ 功能 OK**:删除按钮藏在三点菜单 + 用软删 hook,用户看不见 + 用了等于没用,等于功能不存在
- **后端 / hook 都有但 UI 没绑**很常见 — review code 时除了"功能写没写",还要看"UI 是否真用上"
- **危险操作必须区分严重程度**:软删 vs 硬删两个独立选项 + AlertDialog 强警告"不可恢复",比单一按钮 + 一个 checkbox 更不易误操作
- 文案改 "移除登记" → "**移除登记(保留文件)**" 让两个选项语义清晰对照
