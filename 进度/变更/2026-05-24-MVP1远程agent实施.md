# 2026-05-24 · MVP-1 远程 VPS Agent 实施

## 30 秒摘要

实施 MVP-1 远程 VPS agent 完整功能切片。架构选 **B + F**(Pull Agent + HTTP Long Polling + Tailscale 网络层),类比 GitHub Actions self-hosted runner:

- **DB**:加 `nodes` 表(slug/auth_token_hash/last_seen_at/version/metadata 等)+ `instances.node_id` 外键 → 默认 `local`(id=1)
- **Backend**:管理员节点 CRUD(session auth)+ Agent 端 4 个端点(Bearer token auth: poll/result/stdout/heartbeat)+ executor 派单分流(本地 → sandbox / 远程 → 写 DB 等 agent 拿)+ SSE 走 DB stdout 字段轮询(远程 run)
- **Agent CLI**(独立 Python 单文件 ~400 行):httpx long poll + 调本地 sandbox_runner 跑脚本 + 增量回传 stdout + 心跳线程 + systemd unit
- **Frontend**:`/nodes` 节点管理页 + 实例编辑表单加"节点"下拉 + 一次性 token 显示 + 一键安装命令

**安全设计**:
- 主密钥 `encryption.key` 永远不离开主面板;主面板加密解密后**下发明文** config 给 agent
- agent token bcrypt 存表(类比 password)+ 一节点一 token + 可单独 revoke + 撤销重生成
- agent 端 API 仅 `/api/v1/agent/*` 走 Bearer auth(其它路径仍走 session)

## 关键设计决策(我做的)

### D1 · agent 远程 run 的 SSE 实时日志协议

**问题**:本地 run 时 SSE 从主进程 stdout 流式拿;远程 run 主面板没 stdout pipe。

**选项**:
- A. agent 用 SSE 反向推流(复杂,需 agent → 主面板长连接,WS-like)
- B. **agent 每 1-2s POST `/runs/{id}/stdout` 增量(stream/seq/data)→ 主面板写 `runs.stdout` + LogBroker.publish → SSE 转发(选这个)**
- C. agent 每次只在结束时回传完整 stdout/stderr(失去实时性)

**选 B 的理由**:
1. 协议简单(HTTP POST,沿用 long polling 同款客户端)
2. 复用主面板现有 LogBroker(直接 publish 即可)
3. SSE 客户端代码零修改
4. agent 端可以 batch 多行一起 POST,减少请求频率

**协议**:
```
POST /api/v1/agent/runs/{run_id}/stdout
Authorization: Bearer <token>
{
  "stream": "stdout" | "stderr",
  "lines": ["...", "..."],
  "seq": 17        // agent 端单调递增,主面板可丢弃乱序
}
```

主面板收到 → `log_broker.publish(run_id, stream, line)` 一行一行推 + append 到 `runs.stdout`/stderr。

### D2 · executor 远程派单的"等 agent 拿"机制

**问题**:executor 现在是 sync execute,从 trigger 到结果一气呵成。远程 run 必须把 pending 写 DB 然后立即返回,等 agent 拉走。

**方案**:
- `execute_run(instance_id, ...)` 内查 instance.node_id → 若 != local:
  - 创建 run row(status='pending', host='node:vps-jm')
  - **不等 agent 完成,立即返回 run_id**
  - run 完成是异步:agent POST result endpoint → 主面板更新 run + 触发通知 + 调度 retry(若失败)
- agent 端 poll 取走 task 时主面板把 run.status `pending` → `running`,并标 host

这意味着 `trigger_now` 的语义对远程 / 本地一致(返回 run_id),但实际执行流程分两条路:
- 本地:executor 自己跑(现有路径)
- 远程:executor 早 return,后台异步等 agent 回传 result

注:本次 MVP-1 不做"任务分配 lease + 超时"机制(MVP-2 范围);现假定 agent 总在线。

### D3 · agent token 鉴权策略

- token 格式:`sa_<32 url-safe chars>` ≈ 256 bits 熵
- 存表:`auth_token_hash` = bcrypt(token)(类比密码,可单条 revoke)
- 创建 / regenerate 接口返回明文 token **仅一次**;前端显示后用户必须保存,刷新页面不再展示
- agent 端 `Authorization: Bearer sa_xxxx` → middleware 拿 token → 全表 bcrypt 校验(N=节点数,通常 < 10,可接受)→ 失败 401
- **优化**(留 MVP-2):加 token 前缀(节点 id base32 encoded)→ 直接定位行,免全表扫

### D4 · 路径分流

- `/api/v1/agent/*` → agent_auth middleware(Bearer),CSRF middleware 豁免
- 其它 `/api/v1/*` → 走现有 CurrentUser session middleware,CSRF 仍生效

### D5 · 不做的事(MVP-2 范围)

- ❌ Tailscale 集成(用户后续手动装,代码层零依赖)
- ❌ agent 自动同步 scripts(本次需要用户手动 scp 脚本到 agent 节点 `/opt/signin-agent/scripts/<slug>/`)
- ❌ agent 自动升级(用户手动重装)
- ❌ 心跳超时自动迁移任务(留 MVP-2)
- ❌ 节点上看 CPU/内存(metadata 字段已存,但 UI 不展示)
- ❌ 多节点 RR / Pin 调度(实例固定 node_id,不支持 fanout)
