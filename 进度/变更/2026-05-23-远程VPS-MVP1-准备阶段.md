## 调研稿要点(给后续接手 / 实施 agent 看)

### 6 种架构对比

| # | 方案 | 类比 | 工时 | 评估 |
|---|---|---|---|---|
| A | Agent + WebSocket 长连接 | 哪吒探针 / Portainer Edge | 40-60h | 中等-高 |
| **B** | **Agent + HTTP Long Polling** | **GitHub Actions self-hosted runner** | **25-40h** | 🌟 **推荐** |
| C | 主面板直接 SSH 远程执行 | AWX / Ansible | 15-25h | 快速 MVP 用 |
| D | Reverse SSH Tunnel + C | — | 中等 | NAT 场景叠加 |
| E | Federation 每节点完整后端 | Mastodon | 80+ h | ❌ 过度 |
| F | Tailscale / WireGuard | — | 5min/节点 | ✅ **叠加 B 用** |

### 推荐 B+F 的关键理由

1. GitHub Actions runner 同款 = 工业级范本,bug 早被踩平
2. agent 主动出站 = 不要求 VPS 有公网入站(NAT / Cloudflare 后皆可)
3. 纯 HTTP 无 WS 状态机 + 主面板无连接池压力
4. Bearer Token 鉴权,融入现有 backend HTTPS + middleware
5. agent 直接调 `sandbox_runner.py`(已有,无需重写执行器)
6. Tailscale 5 分钟一节点装好,**零代码**叠加 WireGuard 加密 + NAT 穿透

### 关键安全决策(任何方案都必须)

**主密钥 `encryption.key` 永远不离开主面板**:
- 主面板加密 secret → 下发**明文** cookie/token(走 HTTPS / Tailnet)给 agent
- agent 用完立刻销毁(env / 临时变量)
- 即使 agent 被入侵,只能拿到**当次任务的明文**,其他实例的 cookie 仍安全

## 关键意外 / 用户需澄清

**用户原话**:"这上面应该有一个正在运行的脚本,JM天堂的签到脚本"

**实际**:这台 VPS 是综合代理/监控机,**完全没有 JM 签到脚本**。可能性:
- A) 用户记错了机器,JM 脚本在别的 VPS
- B) JM 签到从未真部署,只是用户记忆中的"准备做的事"
- C) JM 脚本在某个 docker container 但 docker 没装(矛盾)→ 排除

**等用户回复**:JM 脚本到底在哪台机器 / 给新机器信息 / 或者直接走 MVP-1 不管 JM。
