# 代码审计报告 · 2026-05-16

## 总体评价

MVP-3 后端已生产上线 https://jb.aijiaxia.cc,代码质量整体良好:架构清晰、设计稿落地完整(8 表 / 36 端点 / 14 步调度流程 / 11 字段类型 / 13 公共组件 / 11 页面 wireframe 几乎都按设计实现),103 断言覆盖核心路径,异常体系成熟,加密 / 鉴权 / CSRF 三件套到位。短板在于:(1) **子进程沙箱透传 `PYTHONPATH=backend` 让用户脚本可 import `app.core.crypto.get_cipher` 拿密钥**,与 § 5.5 "严禁透传敏感配置"的精神冲突,这是当前最大的安全设计缺陷;(2) **取消运行(cancel_run)只翻 DB 状态,不真终止子进程**;(3) 多处前后端契约小不一致(UpcomingItem/RecentFailureItem 的 `script_name`、RunListItem 的 `instance/script` 嵌套对象),会让 UI 出现空白但不崩;(4) **`instances.next_run_at` 仍未由 scheduler 回写**,导致 dashboard "下次执行" KPI 永远为 null;(5) **`/openapi.json` 在生产无鉴权暴露**所有端点表面。除安全 1 项是 Critical,其余皆 High/Medium,均可在 MVP-4 限定窗口内修完。

## 🔴 Critical(影响安全 / 数据 / 可用性,必修)

1. **子进程 sandbox 可读取主进程加密密钥** · 位置:`backend/app/scheduler/executor.py:136-141` (`env["PYTHONPATH"]=str(_BACKEND_DIR)`) + `backend/app/runner/sandbox.py:88` (把 script_dir 追加到 sys.path) · 修法建议:子进程 PYTHONPATH 应只包含 `backend` 中**runner 包**的最小集合(可拆 `runner/` 出去单独路径),或在 sandbox 入口禁用 `app.core.crypto` / `app.db` 等敏感模块的 import(在 `__import__` hook 里黑名单);最低成本方案是把 manifest 标注 `runtime.isolated=true` 时同时禁止脚本 `import app.*`(sandbox 启动时在 `sys.modules` 注入 sentinel)。同时在 README/进度文档加红色警告"v1 sandbox 不防御恶意脚本作者"。

2. **`/runs/{id}/cancel` 不真终止子进程,只翻 DB 状态** · 位置:`backend/app/services/run_service.py:159` (`cancel_fn = getattr(scheduler, "cancel_run", None)` → None,实际什么也没做) · `backend/app/scheduler/engine.py` 全文未实现 `cancel_run` 方法 · 修法建议:在 `executor.execute_run` 启动子进程后,把 `{run_id → proc}` 注册到 `SchedulerService.active_processes`;`scheduler.cancel_run(run_id)` 调 `_terminate_subprocess(proc)`;run_service.cancel_run 在 scheduler 调用成功后才写 DB status=cancelled,否则 409。当前 bug 的后果:用户点取消后子进程仍在跑,可能继续发签到请求、覆盖 DB 状态、产生 SSE 事件冲突,严重违反"取消"语义。

3. **`MOBILE_BREAKPOINT=480` 让平板/小笔记本无 mobile UI 体验** · 位置:`frontend/src/hooks/use-mobile.tsx`(README 提到改为 480)· 修法建议:shadcn 默认 768 是 Tailwind `md:` 断点,把它改 480 让平板永远走 desktop 布局会导致 800px iPad / 笔记本侧栏占满主区域。建议改回 768,把"侧栏覆盖 vs push"控制逻辑挪到 `<AppSidebar>` 内部(根据宽度选 sheet 或 push 模式),保留 < 480 真手机仍走 sheet,480-768 用 push,>= 768 用持久 sidebar。

## 🟠 High(影响功能 / 体验,应尽快修)

4. **`instances.next_run_at` 字段永远为 null** · 位置:`backend/app/scheduler/engine.py:_register_inner` 没回写 DB · 影响:`/dashboard/overview` 的 `next_run_at` 字段(MVP-3A 新增)永远 null,`/dashboard/upcoming` 也总是空,KPI "下次执行" 卡片永远显示 `—`,真实生产体验失分严重 · 修法建议:`_register_inner` add_job 之后用 `self._job_next_run(job_id)` 拿到 next 写回 instance 表(单 session 写一次即可);或在 cron job 触发时由 `_scheduled_job_runner` 写回 instance.next_run_at = scheduler.get_job(jid).next_run_time。

5. **前端 Dashboard 期望的 `script_name` 字段后端不返回** · 位置:`frontend/src/api/mocks/dashboard.mocks.ts:60` (`script_name: string`) + `frontend/src/pages/dashboard/Dashboard.tsx:293/356` (`{it.script_name}`) vs `backend/app/schemas/dashboard.py:UpcomingItem/RecentFailureItem`(只有 `script_slug`)· 影响:即将执行 / 最近失败两栏在真后端数据下文本是 undefined 渲染为空,只剩 instance_name · 修法建议:后端 `dashboard_service.get_upcoming` / `get_recent_failures` 各加一列 join Script 拉 name,RecentFailureItem schema 加 `script_name: str`;前端类型同步;OR 前端用 `script_name ?? script_slug` 容错。这是 ADR-014 提到的"契约对齐"问题的二次复发。

6. **前端 InstanceListItem / RunListItem 用嵌套对象,后端只返 slug/id** · 位置:`frontend/src/api/hooks/instances.ts:41-63` + `frontend/src/api/hooks/runs.ts:41-66` · 影响:RunList / InstanceList 列表中可能依赖 `run.instance?.name` / `run.script?.name` 显示但实际是 undefined,需要客户端再发一次 N+1 请求才能渲染完整 · 修法建议:后端 RunListItem schema 加 `instance_name: str | None` + `script_name: str | None`(join scripts/instances 各一次,加 cache);或前端列表只展示 `instance_id`/`script_slug` 不假设 name 可用。

7. **N+1 查询:GET `/scripts` 列表每条调一次 `_count_instances_for`** · 位置:`backend/app/api/v1/scripts.py:81` · 影响:页面 size=20 时发 20 次 SELECT COUNT,即使 SQLite WAL 也是不必要开销;且引用 `_` 开头私有函数破坏封装 · 修法建议:`script_service.list_scripts` 用 `select(Script, func.count(Instance.id))` outerjoin + group by 一次拉,把 instance_count 直接拼到 ListItem。

8. **`disable_script` 不停 scheduler job,导致后台调度噪音** · 位置:`backend/app/services/script_service.py:disable_script:314` 不调 `scheduler.unregister`,scheduler 仍按 cron 触发 → executor 在 [2] 步检查 `script.enabled` 后跳过 · 影响:不致命(脚本不会真跑),但产生无效日志、占 scheduler 槽位、被禁用脚本仍可能命中 misfire grace;违反设计稿 § 2.2 "禁用 script 会暂停其所有实例的调度但不删" · 修法建议:`disable_script` 接受 `scheduler` 参数,对该 script 所有 instance 调 `scheduler.pause(instance_id, until=远未来)`;`enable_script` 时调 `scheduler.resume`。

9. **`/openapi.json` / `/docs` 在生产无鉴权暴露** · 位置:`backend/app/config.py:expose_docs=True` 默认 + `deploy/nginx-jb.aijiaxia.cc.conf:77` 透传出来 · 影响:任何人 curl `https://jb.aijiaxia.cc/openapi.json` 可枚举全部 36 端点的入参 schema,辅助攻击者构造精确请求;暴露面没必要 · 修法建议:生产 `.env` 显式 `expose_docs=false`(代码已支持),或 nginx 加 `location = /openapi.json { return 404; }`;若仍想保留供前端类型生成,改用 docker exec 进容器 curl localhost。

10. **`PATCH /instances/{id}` 没有 cron_expr 合法性预校验** · 位置:`backend/app/services/instance_service.py:update_instance:343` 直接赋值,在 `scheduler.reschedule` 抛 cron 解析错误时已存了 DB · 影响:用户改了非法 cron,API 看似成功(DB 改了),但 scheduler 在下次 register 时报错,且永久卡住 · 修法建议:update_instance 在赋值前对新 cron 显式 `CronTrigger.from_crontab(new_cron, timezone=...)` 校验;非法则抛 `CronExprInvalidError` 422。同 issue 也存在于 InstanceCreate(只在 schema 层做了 max_length=64,没做 cron 语义校验)。

11. **`response_model_by_alias` 默认 True 与 `secret_set` alias 的不显式约定** · 位置:`backend/app/schemas/instance.py:212-216` 用 `alias="_secret_set"` 但 model_config 未声明 `populate_by_name=True` · 影响:FastAPI 默认 `response_model_by_alias=True` 工作正常,但若后续某个 agent 改 InstanceDetail 用 `model_validate(dict)` 后调 `model_dump(by_alias=False)`,前端会拿到 `secret_set`(无下划线)→ 解 `_secret_set` 显示 undefined · 修法建议:加 `model_config = ConfigDict(from_attributes=True, populate_by_name=True)`,并在 FastAPI 装饰器显式 `response_model_by_alias=True` 自文档化。

12. **`get_scheduler` 引发 RuntimeError 时被 error_handler 兜成 500 而非 503** · 位置:`backend/app/deps.py:115` `raise RuntimeError("Scheduler 尚未初始化")` · 影响:启动期或 shutdown 中所有调度相关端点(POST /instances/run、cancel)返 500,日志会污染 ERROR 级别 · 修法建议:用 `app.core.exceptions.AppException` 子类 `SchedulerNotReady(status_code=503)`,统一日志走 INFO 级别。

13. **前端 `RunsFilter.order` 参数后端 list_runs 不接受,会被忽略静默失效** · 位置:`frontend/src/api/hooks/runs.ts:76` (`order?: 'asc' | 'desc'`) vs `backend/app/api/v1/runs.py:91-150`(只接 instance_id/script_slug/run_status 等) · 影响:前端若传 `order='asc'` 期望升序但仍是 DESC,排序需求被悄悄忽略 · 修法建议:后端 list_runs 加 `order` Query 参数,或前端从类型里删掉这个字段。

14. **设计稿 § 1.6 "同 channel 命中多 rule 取 scope 更具体的"实现走的是 priority + id 大者** · 位置:`backend/app/notifications/dispatcher.py:104-113` · 影响:设计稿 "instance > script > global" 由 priority 实现没问题,但 "同 priority 取 id 大" 是新增语义(设计稿未规定),会让"用户创建顺序"间接决定通知去向 · 修法建议:加 ADR 记录"同优先级取后创建者覆盖"或改为"先创建者优先"(更稳定);更要紧的是补单元测试 fixture 锁定行为。

15. **`backup/import` 仅解析 meta,实际无效** · 位置:`backend/app/api/v1/settings.py:240-304` · 影响:用户上传 zip 后看到 "v1 暂不自动替换"消息,但 API 返 200 OK,体验上像成功;若用户后续没看 message 字段,以为已恢复 · 修法建议:短期把响应 status 改 202 Accepted + 红色提示文本;长期实现真替换(写 `data/backups/before-restore-{ts}/` + alembic check + 触发 `os.execv(uvicorn)` 自重启)。

## 安全 / 性能 / UX 各维度评级

### 安全:**C+**

**好的地方**:
- Fernet 字段级加密 + MultiFernet rotation 预留(`core/crypto.py`)
- bcrypt rounds=12 + secrets.token_urlsafe(48) ≈ 256-bit 熵
- HttpOnly + Secure(prod) + SameSite=Lax cookie 三件套
- CSRF 二保险(X-Requested-With + SameSite)
- 用户不存在 vs 密码错误统一 "InvalidCredentials" + 一次假 bcrypt 防侧信道
- 失败 5 次锁 15min(防爆破)
- secret 字段 PATCH 未提交 → 保留旧值的"语义合约"完整(fields.merge_secrets)
- error_handler 生产环境兜底 500 + trace_id,不暴露 traceback

**扣分点**:
- **Critical #1** 子进程可 import 主程序 → 密钥可读
- subprocess env 最小集白名单做了,但 PYTHONPATH=backend 让保护失效
- `/openapi.json` 生产无鉴权暴露(High #9)
- backup/export 含 key 的请求只用 `?include_key=true` 区分,没用 confirmation 文字 / 二次密码确认
- Fernet 主密钥在生产服务器明文 file (`0o600`,但 root 可读) — 设计稿已承认 v1 trade-off
- bcrypt rounds=12 写死,无 settings 表项,2026 末可能想升到 13

### 性能:**B**

**好的地方**:
- SQLite WAL + busy_timeout 5000 + 20MB cache 全应用(`db/pragma.py`)
- runs 表 4 个复合索引(instance_started/started_at/status_started/script_started)
- log_broker per-run channel + max_subscribers=10 + queue_size=2048 防慢消费者
- frontend bundle vite manualChunks 拆 12 个 vendor,主 bundle 2095KB→252KB
- TanStack Query 5 缓存 staleTime 15-30s
- executor 子进程 stream pump 实时推 broker(不积压)
- AppriseClientPool 单例 + asyncio.Lock 避免重建 + asyncio.to_thread 避免堵 loop

**扣分点**:
- **High #7** GET /scripts 列表 N+1(N 条调 N 次 COUNT instances)
- session 每次请求写 last_used_at,dashboard 30s 自 refetch 5 端点 = 5/min UPDATE
- `_count_notifications_24h` try/except 兜底太宽
- `dashboard_service.get_timeline` SQLite-only 实现(strftime),迁 PostgreSQL 要重写
- `disable_script` 不停 scheduler job → 调度噪音
- `_register_inner` 每个 instance 进函数若无 cron 就再开 session 拉 script(浪费)
- vendor-time 302KB(luxon + cron-parser 5.x 联动,可考虑 dayjs 替换)

### UX:**B+**

**好的地方**:
- 13 个公共组件齐全(KpiCard/Sparkline/ScriptCard/Timeline/SecretInput/CronInput/DynamicForm/LogViewer/InstanceFormSheet/InstancesPanel/RunsPanel/RunDetailSheet/CommandPalette/EmptyState/PageHeader/StatusBadge/DataTable/RouteErrorBoundary 17 个,超出设计稿 13 个)
- shadcn/ui + Tailwind v4 + OKLCH CSS vars + 浅深色一等公民 + 主题色 picker(react-colorful)
- LogViewer xterm 全套:暂停/全屏/搜索/导出 + 主题热刷 + autoFollow 智能跳底部
- DynamicForm 11 字段类型映射 11 种控件 + secret 字段"未变更 = 不提交"语义
- Timeline + Sparkline + Recharts + Framer Motion 动效
- ⌘K cmdk 命令面板挂在 AppLayout
- RouteErrorBoundary 兜底 + nginx index.html no-cache 双保险(MVP-3 hotfix)
- Sonner toast + AlertDialog 二次确认(删除/暂停)
- "实时跟随" Switch / 状态多选 / DateRangePicker + 4 个 preset

**扣分点**:
- **High #5** Dashboard `script_name` 字段后端不返,显示空白
- **Critical #3** MOBILE_BREAKPOINT=480 让 480-768 平板宽度走 desktop 拥挤布局
- "下次执行" KPI 永显 `—`(High #4)
- 主密钥生成警告只在日志,UI 没红色横幅
- backup/import 200 OK 但实际没还原,只有 message 提示用户(High #15)
- 错误详情 stack 在生产 minified bundle 显示乱码(Low #42)
- 部分 toast 文本不一致(`已启用` vs `已禁用` vs `已删除`)

### 工程化:**C+**

**好的地方**:
- 进度文档结构清晰(README/决策/分支/变更/设计)
